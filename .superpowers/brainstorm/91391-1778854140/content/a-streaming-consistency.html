一致性原则（写代码时遵守）

  • 容器先定，内容后填 — block 一被 parser 识别出类型，立刻渲染最终容器（左竖线 / 圆点 / pill 形态全到位），随后只追加文本和按钮。
  • icon 同尺寸同位置 — spinner 12×12 → check 12×12，绝对不允许 reflow。
  • 选项 pending 态用 dashed — option 按钮的 JSON 还没解出来时，用同尺寸 dashed 灰色占位（不是骨架条不是 shimmer）。
  • code-head 始终在 — fence 一开 lang 标签就出现，copy 按钮在完成后追加（位置预留好）。
  • post-hoc 渲染不预占 — draft / inferred / invalid 是事后扫描，流式中绝不出现占位；流完一次性 append 在 prose 末尾（不替换/包裹原内容）。
现实现里要砍掉的反模式：
· StreamingTalorSkeleton（animate-pulse + 灰条）→ 改为最终容器 + dashed pending option
· if (!isStreaming) renderInferredIntent → 流式中本来就不渲染，逻辑保留但视觉无跳变（因为之前也没东西）
· StreamingTalorSkeleton 切换到 TalorBlockCard 的"换皮" → 改为同一组件，仅 data-streaming 属性不同
· 流式中 prose-pre 边框/底色跟完成态不一致 → 容器从第一行 fence 就保持终态
· "已复制 2s 后变回 复制" 这类 transient 状态：保持 button 尺寸不变，仅文字替换。

这套一致性规则把所有反例（skeleton → card 跳变、post-hoc 卡突然出现、icon reflow）都堵上了。要不要把全部 4 份 spec（场景全集 + markdown + block 弱化 + 流式一致性）合并成一份 design doc，下一步走 writing-plans？
