优势：
· Talor 不知道 email 概念 — 完全 MCP-agnostic
· 任何 MCP 工具开箱即用，无须改 UI 代码
· LLM 决定哪些字段相关 / 怎么呈现 — 比写死 schema 更聪明
· UI 只维护一套 markdown 渲染器 — 维护成本最低
什么 UI 该 Talor 自己写、什么交给 LLM
类别 谁负责渲染 原因
内置 7 个工具
bash / edit / read / grep / write / glob / ls 		Talor 写专门渲染器 输出结构固定（diff / 命中行 / terminal output），渲染语义有产品差异化价值
Tool row（任何工具） Talor spinner / check / error / 工具名 / 摘要 / 耗时 — 这是工具调用本身的元数据，跟工具语义无关
MCP 工具的内容输出
gmail / calendar / web_search / ... 		LLM 通过 markdown LLM 看 JSON → 决定哪些字段相关 → 输出 markdown 表/列表/段落。UI 只渲染 markdown
展开看原始 JSON
debug 用 		Talor（折叠） tool row 可点开看完整 raw output，方便调试
交互动作
"发送" / "保存" / "选项" 		need_input / draft_detected block 这两个是 Talor 已有的结构化 block — LLM 想给用户提供按钮就发 need_input；想让用户保存某个对象就发 draft_detected。不需要新机制
最终设计原则
  • UI 的"业务知识"仅限于内置 7 个工具。其他全 MCP-agnostic。
  • 每个 MCP tool 在 UI 里就是：✓ tool_name · 摘要 · 耗时 [▸ JSON]
  • "摘要"由 MCP server 在 tool result 里提供（如 "3 results"），或 LLM 在 input 时设置 hint。无的话退化为"output size 字符数"。
  • LLM 看完 JSON，prose 里用 markdown table / list / blockquote 把关键信息说出来。
  • 交互按钮统一走现有的 need_input / draft_detected block — 不新增按钮机制。
代价：方案 A 里的 email card / calendar event card / draft email card 这些专属 UI 不存在了，全部退化为 markdown 表格 + 链接。视觉丰富度下降，但换来 MCP 生态的完全开放。
