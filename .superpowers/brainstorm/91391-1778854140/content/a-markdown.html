Talor 的 ReactLoop 死循环保护机制

本文解释 src/main/loop/react-loop.ts 里的核心控制流，以及它如何在不引入显式 retry 计数器的情况下，依然保证不会失控。

1. 整体流程

主循环每一轮做三件事 — 拉 messages、调 provider、跑工具。三步任意一步抛错都会终止 turn，把 envelope 写回 db 后冒泡到 IPC。

循环上限有两层：

  1. 硬上限 — maxIterations 默认 20，越界直接 throw
  2. 软上限 — context window 接近耗尽时调 ShortTermMemory.compact() 主动压缩

1.1 step_id 漂移检测

每个 tool_use 必须携带一个递增 step_id。重复 id 会让 loop 立刻 halt，把当前进度落盘后退出。

这条规则不靠 prompt 软引导，而是在代码里强制 — SeenStepIds 是个 Set，has() 命中即 throw。

详见 standards.md §J-SHOULD-1。

关键参数

参数 默认值 可调 作用
maxIterations 20 ✓ 硬上限，超过即抛 CONTEXT_HALT
contextWindow 200,000 ✓ 触发 ShortTermMemory 压缩的阈值
stepIdCheck on − halt 检测，不允许关闭
compactRatio 0.6 ✓ 压缩后保留的内容比例

2. 代码骨架

简化后的循环体长这样：

typescriptreact-loop.ts:128-156copy 
async function runLoop(ctx: LoopCtx): Promise<LoopResult> {
  const seenIds = new Set<string>()

  for (let iter = 0; iter < ctx.maxIterations; iter++) {
    const messages = await ctx.pipeline.build()
    const step = await ctx.provider.generate(messages)

    if (seenIds.has(step.id)) {
      throw new ContextHaltError(step.id)  // 漂移即 halt
    }
    seenIds.add(step.id)

    if (step.kind === 'done') return step
    await ctx.tools.run(step.toolCalls)  // 结果通过事务回写 db
  }
  throw new MaxIterError(ctx.maxIterations)
}
