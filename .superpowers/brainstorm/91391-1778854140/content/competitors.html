Talor 该放哪里 · 分析
  • Manus / Devin 路线不适合 Talor — 这两家依赖 computer-use（截屏 + 鼠标 + 实时 GUI）。Talor 的 MCP 工具返回 JSON，没"画面"可看 — 强行做 split pane 右侧没内容。
  • ChatGPT 路线扩展性差 — 每种工具硬编码 UI，把生态封闭。我们之前否决的"注册表"方案就是这条路。
  • Cursor 路线适合代码场景但 Talor 是通用 agent，邮件 / 表格 / 报表也要好看。
  • Claude.ai 的 Artifact 思路值得借鉴 — 长文档 / 长代码 / 多步骤计划这类"富内容"可考虑单独 pane（但 v1 不做，先做好 inline）。
  • v0 generative UI 跟我们的 block 方案最接近 — 都是"LLM 在小套预定义组件里挑"。Talor 的 5 个 block + markdown 就是它的 mini 版。
Talor 跟 v0 的差异
  • v0 组件库针对业务（Stock / Weather / Flight）— 跟特定场景绑死，要扩展须发布新组件
  • Talor block 是交互模式而非业务（need_input = 让用户选 / proposal = 让用户确认）— 任何业务场景都用同一套 block，配合 markdown 表达内容
  • v0 适合明确的垂直域产品（StockGPT、TravelGPT），Talor 是通用平台，必须避免业务耦合
所以 Talor 不该照搬 v0 的组件库思路，而是把 block 词典做得小而抽象（5-6 个），把表达力交给 markdown + LLM。
v2 可以考虑的增强
  • Artifact 面板（Claude.ai 风）— agent 生成长 markdown / 代码 / 报表时单独占右侧 pane。v1 暂不做（inline 够用）。
  • Plan tab（Devin 风）— 长任务时左侧加一个 plan tab，显示步骤列表 + 进度。Talor 当前 agent 任务多是单 turn 完成，v2 多步骤复杂任务上线时考虑。
  • "computer use" via MCP（Manus 风）— 如果加 browser-use MCP server，需要右侧 pane 显示截图。当前不是主路径。

小结：Talor 当前方向（inline + 5-6 个抽象 block + markdown）跟 v0 generative UI 最像，但更抽象 / 跨域。v2 可考虑借鉴 Claude.ai 的 artifact 面板。
