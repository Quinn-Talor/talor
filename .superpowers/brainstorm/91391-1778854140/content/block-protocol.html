ASchema 定义（单一可信源）

每个 block 在 src/shared/ui-rendering/blocks.ts 用 Zod 定义。这是 LLM 输出和 UI 渲染的同一份 truth。 

// src/shared/ui-rendering/blocks.ts
import { z } from 'zod'

// ─── need_input ─────────────────────────────────────
export const NeedInputSchema = z.object({
  type: z.literal('need_input'),
  question: z.string().min(1),
  options: z.array(z.string()).min(2).max(5),    // 强制 2-5 个
  reason: z.string().optional(),
})

// ─── proposal ───────────────────────────────────────
export const ProposalSchema = z.object({
  type: z.literal('proposal'),
  summary: z.string().min(1),
  preview: z.string().optional(),       // markdown 允许
  action: z.object({
    label: z.string().min(1),
    tool: z.string().min(1),         // 必须是 registry 注册的 tool
    args: z.record(z.unknown()),
  }),
  secondary_actions: z.array(z.object({
    label: z.string(),
    emit: z.string(),                  // 塞回 LLM 上下文的指令
  })).optional(),
})

// ─── done / blocked / warning 同模式 ───────────────
export const TalorBlock = z.discriminatedUnion('type', [
  NeedInputSchema,
  ProposalSchema,
  DoneSchema,
  BlockedSchema,
  WarningSchema,
])

export type TalorBlock = z.infer<typeof TalorBlock>
5 个 block 字段表
Block 必填 可选
done summary metrics: {tools, duration_ms, files_modified, ...}
need_input question · options (2-5) reason
blocked reason retry_hint · retry_tool+retry_args
warning message · severity (low/medium/high) code · tool_intercepted
proposal summary · action{label, tool, args} preview · secondary_actions[]
BSystem prompt 注入（LLM 怎么"学会"用 block）

在系统 prompt 末尾固定追加一段 block protocol（由 UiBlockPlugin 注入，跟其他 plugin 合并）。例： 

--- (system prompt block section) ---

You can emit structured UI blocks in your response. Use them sparingly —
markdown is the default; blocks are for interaction or terminal state.

Each block is a fenced code block with language `talor`:

```talor
{"type": "<block-type>", ...}
```

### need_input  — Ask user to pick from 2-5 options
Use when you need a discrete choice (not open-ended). Don't use for confirmations
(use `proposal` for those).

```talor
{
  "type": "need_input",
  "question": "Which time slot works?",
  "options": ["Mon 10:30", "Tue 15:00", "Let me check others"],
  "reason": "Each slot has different conflicts."
}
```

### proposal  — Propose a one-click action user can confirm
Use when you've prepared something the user should review and confirm with one click
(send email, save config, create event, ...). The action's `tool` must be a
real tool name; `args` must satisfy that tool's schema.

```talor
{
  "type": "proposal",
  "summary": "Email draft to wang@acme.com - Re: Q4 renewal",
  "preview": "Hi Wang,\n\nRe: your two questions...",
  "action": {
    "label": "Send",
    "tool": "gmail.send_draft",
    "args": {"draft_id": "abc-123"}
  }
}
```

### done / blocked / warning  — Same fenced format with their schemas
(...full docs...)

--- (end block section) ---
关键：prompt 里给 schema + 1-2 个真实例子 + 何时用。LLM（特别是 Claude / GPT-4 级）能稳定输出符合 schema 的 JSON。Talor 已有的 need_input 实践证明：4 句话 + 1 个例子，命中率 95%+。

D流式 parser 识别（已有，需扩展）

Talor 已有 splitMessageWithTalorBlocks 流式 parser（在 @shared/ui-rendering/）。扩展点：

  • 识别 ```talor 开 fence → 切到 block 模式
  • 边流入边 buffer，前 20 byte 解出 type 字段 → 立刻渲染最终容器（per 流式一致性规则）
  • fence 关闭后用 Zod schema 校验 — 通过则渲染填充内容，失败则降级为 invalid（不渲染 UI，进开发者 log）
  • 未知 type 字段（不在 5 个 block 之内）→ 当作 invalid
// pseudo-code
function parseStreamingTalorBlock(raw: string): ParseResult {
  const typeMatch = raw.match(/"type"\s*:\s*"(\w+)"/)
  if (typeMatch) return {
    phase: 'streaming',
    type: typeMatch[1],    // 已知 → render container 立刻定型
    partialBuffer: raw,
  }

  if (isFenceClosed(raw)) {
    const parsed = TalorBlock.safeParse(JSON.parse(raw))
    return parsed.success
      ? { phase: 'done', block: parsed.data }
      : { phase: 'invalid', raw, error: parsed.error }
  }
}
E渲染 + 用户点击

渲染分发用单一 switch：

function TalorBlockRenderer({ block, onAction, onEmit }) {
  switch (block.type) {
    case 'need_input':
      return <NeedInputBlock {...block} onPick={onEmit} />
    case 'proposal':
      return <ProposalBlock {...block} onConfirm={onAction} onAlt={onEmit} />
    case 'done':
      return <DonePill {...block} />
    case 'blocked':
      return <BlockedRow {...block} onRetry={onAction} />
    case 'warning':
      return <WarningRow {...block} />
  }
}
用户点击行为
用户动作 系统反应
need_input → 点选项 "方案 A" 把字符串 "方案 A" 当作新 user message 发回 LLM
proposal → 点 "发送" CTA 调 toolRegistry.invoke(action.tool, action.args)，结果作为 tool_result 入 LLM 上下文
proposal → 点 secondary "编辑" 把 secondary_actions[].emit 字符串当作新 user message 发回 LLM（LLM 进入修改流程）
blocked → 点 "重试 ↻" 若有 retry_tool 则调用；否则简单发"重试"消息
warning → （无交互） —
Fround-trip 安全门

用户点 proposal 的 CTA 时，不直接信任 LLM 给的 tool + args。Talor runtime 做三道校验：

  1. Tool 存在 — action.tool 必须是 registry 注册过的（内置 7 个或当前 agent 启用的 MCP server tools）
  2. Args 校验 — 用该 tool 的 Zod schema 校验 action.args
  3. 权限检查 — 走现有 PermissionGuard（如 bash 命令黑名单 / 路径白名单 / MCP server enabled 状态）

任意一道失败：UI 不调工具，把错误 envelope 作为系统消息塞回 LLM 上下文（"Tried to invoke X but got DENIED because Y"），让 LLM 解释 / 改提议。

为什么必要：LLM 可能幻觉出不存在的工具名 / 编造 args，或者用户的 agent profile 本来就没赋权某 tool。这层校验是不可绕过的，把 LLM 的"提议"权限和 Talor 的"执行"权限分开。
G渲染时序（一并解决流式一致性）
--- LLM 流式输出 ---
看了下 SLA 文档，顺手起了个草稿：

```talor
{"type":"propos                ← parser 还在 buffer，UI 不渲染
{"type":"proposal","summary":"   ← type=proposal 已知，UI 立刻渲染容器（左竖紫线 + 占位 summary）
{"type":"proposal","summary":"邮件草稿..." ← summary 流入
"preview":"王总你好                  ← preview 流入
"action":{"label":"发送"             ← action 容器内的 button 出现，但 disabled（args 还没解完）
"args":{"draft_id":"draft_8f"}} ← args 完整 → CTA 变 enabled
```                              ← fence 关 → Zod 校验 → 终态

发送前你看看...             ← prose 续

整个过程 UI 容器形态在 ~20 byte 后就稳定（左竖紫线 + summary 区 + preview 区 + action 区位置全到位）。文本和 CTA enabled/disabled 在内部填充，不会发生 skeleton → card 的跳变。

✓核心机制总结
  1. 5 个 block 的 Zod schema 是 LLM 输出和 UI 渲染的单一可信源（@shared/ui-rendering/blocks.ts）
  2. System prompt 教会 LLM 何时用 + 怎么填（schema + 例子 + when-to-use）
  3. LLM 在合适时刻自主输出 block（嵌在 markdown 文本中，用 ```talor fenced 包裹）
  4. 流式 parser 识别 type 后立刻渲染容器，内容流入填充（无跳变）
  5. 用户操作回环：选项 → 新 user message · CTA → 调工具 · 二级 action → 触发 LLM 子流程
  6. Tool 调用前三道校验（存在 / args / 权限），LLM 提议但 Talor 执行

这块逻辑要不要加到 spec？建议作为 §11A · Block 协议（在 §11 Talor block 渲染之前），把 schema / prompt 注入 / parser / 安全门一次性说清楚，便于实现。
