LLM 判断渲染 · 通用 block 抽象

UI 不认识 email / calendar / web，只认识一小套通用 block。LLM 看 tool 输出后选择用什么 block 呈现。Talor 完全 MCP-agnostic。

1通用 block 词典 · LLM 的"渲染原语"

Talor 已有 5 个 block，加一个 proposal（泛化版 draft_detected）就足够。UI 完全不认识 email / calendar / web，只认识这 6 个 block 类型 + markdown。

Block 用途 状态
done turn 结束 · 摘要 已有
need_input 问问题 · 提供 2-5 个选项让用户点 已有
blocked 跑不下去了 · 让用户处理某事后 retry 已有
warning 提醒一件事（非阻塞） 已有
proposal改名 提议一个用户可一键确认的动作（发邮件 / 保存 agent / 创建会议 ...） 由 draft_detected 泛化

所有"邮件 / 日历 / 网搜 / 报表"等业务概念都通过 markdown（table / list / blockquote / heading）+ 上述 block 表达。LLM 看到 tool JSON 输出后，自主决定：

  • 展示数据用 markdown table / list
  • 需要用户做选择题用 need_input
  • 需要用户确认一个动作（不可逆 / 重要）用 proposal
  • 需要提醒用 warning / blocked
2proposal block · schema
{
  "type": "proposal",
  "summary": "邮件草稿 · 王总（Acme） · Q4 续约方案",  // 一行摘要
  "preview": "王总你好，\n\n收到你的来信...",            // markdown preview，可滚动 / 截断
  "action": {
    "label": "发送",
    "tool": "gmail.send_draft",                       // 任意 MCP / 内置工具
    "args": { "draft_id": "abc-123" }                  // 工具参数
  },
  "secondary_actions": [                                // 可选
    { "label": "编辑", "emit": "edit_draft" },        // 触发 LLM 再次响应
    { "label": "重写", "emit": "rewrite_draft" }
  ]
}

设计要点

  • summary 是必填 — 单行概括"将要发生什么"
  • preview 可选 — 给用户看完整内容（markdown，可滚动）
  • action.tool 是任意 tool name — 可以是 MCP 工具、可以是内置工具。UI 不关心，只负责"用户点了就调用"
  • secondary_actions 用 emit — 不直接调工具，而是把字符串塞回 LLM 上下文（让 LLM 进入修改流程）
  • UI 渲染恒定 — 不论 action 是发邮件还是保存 agent，UI 都是"2px 紫左竖线 + summary + preview + CTA"

这个 UI 渲染的什么

  • tool row · 通用
  • markdown 表格 · 通用
  • need_input · 通用
  • proposal · 通用（这里 action.tool 是 gmail.send_draft，但 UI 不知道）

4三种方案对比
A · 注册表硬编码 B · 纯 markdown C · LLM 选 block（推荐）
业务知识 UI 知道邮件 / 日历 / 网搜 UI 不知道 UI 不知道
新 MCP 工具 要改 UI 代码 开箱即用 开箱即用
交互动作（按钮） 每种工具单独写 只有链接，无 CTA proposal block 通用
视觉丰富度 最丰富 最简朴（仅 markdown） markdown + 5-6 个 block，够用

核心机制：扩展 Talor 已有的 block 系统 —— 把 draft_detected 改成通用 proposal（任意 tool + args + label），LLM 自主选择。UI 渲染 6 个 block + markdown，完全不认识业务概念。
