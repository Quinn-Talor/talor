# Talor v3.7.1 — LLM × 系统 协作模型 + 职责边界清理

> 起草日期: 2026-05-13
> 状态: 已实施
> 前置: v3.7 已落地 ([2026-05-13-talor-v3.7-fault-tolerance-rebalance.md](2026-05-13-talor-v3.7-fault-tolerance-rebalance.md))
> 目标: 把 v3.7 的"信任 LLM"原则**外化为正式协作矩阵**,作为后续所有架构决策的依据;并清理 v3.7 残留的职责越界点

---

## 0. TL;DR

v3.7 删了 forced-closure 灾难路径,但 v3.7.1 进一步追问:**LLM 和系统的边界到底在哪**?

**核心原则**(写入 standards.md):

- **LLM 擅长**: 推理 / 判断 / 自然语言 / 语义理解
- **系统擅长**: 执行 / 校验 / 兜底 / 审计 / 事实核对(基于运行时真相数据)

**两种典型职责越界**(本次清理目标):

- **系统抢 LLM 活** — 用 regex 做语义判断,误差大且只软建议 → 两边不靠岸
- **LLM 担系统责任** — 强制 LLM 守 streaming 解析便利的约定 → 增加 LLM 负担,反而降低遵从度

## 2. 决策点责任矩阵 (Canonical Reference)

本表是后续所有"该谁做"争议的标准答案。

| 决策点              | LLM 主责                     | 系统主责                          | 协作交接                                                                   |
| ------------------- | ---------------------------- | --------------------------------- | -------------------------------------------------------------------------- |
| **该不该调工具**    | ✅ 推理决定                  | —                                 | LLM 输出 tool_call,系统 execute                                            |
| **调哪个工具**      | ✅ 选择                      | —                                 | system prompt 提供工具列表,LLM 选                                          |
| **工具参数生成**    | ✅ 生成                      | 校验(Zod / path-guard)            | LLM 写 input → 系统校验 → 失败返结构化错误                                 |
| **工具能否执行**    | —                            | ✅ 校验 / 拦截                    | 校验失败 → `ToolErrorEnvelope` 回流 LLM 自适应                             |
| **工具失败重试**    | ✅ 决定 (读 envelope)        | ✅ 死循环兜底                     | LLM 自适应,N 次失败 → forced summary 终止                                  |
| **声明副作用**      | ✅ emit `pending_confirm`    | ✅ fallback regex 兜底            | LLM 主路径声明,LLM 忘则系统兜底拦危险关键字                                |
| **副作用授权决策**  | —                            | ✅ 弹 confirm + 记忆              | 系统弹 UI → 用户决定 → `SessionApprovalMemory` + `Ledger`                  |
| **该不该终止 turn** | ✅ **不发 tool_call 即终止** | —                                 | "无 tool" = "LLM 决定结束",**系统不强制纠正**                              |
| **任务完成判断**    | ✅ 推理决定                  | —                                 | LLM 自然语言表 done,系统不二次判断                                         |
| **需要用户输入**    | ✅ 自然语言询问              | —                                 | LLM 写问题 + 不调 tool,系统识别为 turn 结束                                |
| **任务被卡住**      | ✅ 自然语言说明              | —                                 | LLM 说"无法做",系统不强制 marker                                           |
| **UI 卡片样式**     | 可选 (emit talor block)      | ✅ 推断兜底 (`inferIntent`)       | LLM 主动 emit → 显式卡片;不 emit → 系统推断卡片样式(**仅 UI,不回馈 loop**) |
| **事实引用准确性**  | 推理                         | ✅ **核对** (`verifyQuotedFacts`) | LLM 写引用,系统对照 tool output 真相,假引用 mask                           |
| **Context 预算**    | —                            | ✅ 监控 + halt                    | 系统估算 token,≥100% 阻断                                                  |
| **Memory 压缩**     | —                            | ✅ 全权                           | 系统压缩历史,LLM 不感知                                                    |
| **跨 turn 状态**    | —                            | ✅ 持久化                         | session/message repo,LLM 每 turn 重读                                      |

│ 8. RiskGate.gate() → confirmTool() 弹 UI           │

  │

## 4. 异常与兜底协作规则

按"主路径 LLM 自适应 / 兜底层系统硬阻断"分类:

| 异常                         | 主路径(LLM 处理)         | 兜底层(系统强制)                                                 |
| ---------------------------- | ------------------------ | ---------------------------------------------------------------- |
| 工具参数错                   | LLM 读 Zod 错误 → 自修   | N 次后死循环 detector 强制 forced summary                        |
| 工具调用失败                 | LLM 读 envelope → 换策略 | `failure-streak` 3 次后 forced summary                           |
| 同 input 重复调用            | LLM 应注意               | `signature-dead-loop` 2 次后 forced summary                      |
| 模型连续调工具无文本         | LLM 应给用户解释         | `tool-only-loop` N 步后 forced summary                           |
| 模型整轮空文本               | (异常)                   | `FALLBACK_SUMMARY` 兜底说点什么                                  |
| 副作用没声明 pending_confirm | LLM 应该声明             | RiskGate fallback regex 检测危险关键字 → 弹无 summary 的 confirm |
| 模型说"用户已确认"但实际没批 | (LLM 错)                 | RiskGate 不读自述,只读真实 memory → **结构上无法欺骗**           |
| Context 快满                 | (LLM 不感知)             | 系统估算 → 软警告 hint;≥100% 硬阻断                              |
| Provider crash / 网络断      | —                        | `classifyLlmError` → `ChatErrorCode` → 前端差异化 UX             |

**关键观察**: 兜底全在"代码执行错误"(维度 A)范畴。LLM 的判断错(维度 B),v3.7.1 后**不主动纠正**,只在能导致**真实危害**(副作用越权)时由 RiskGate 拦。

## 5. 反模式识别表

代码 review 时,以下模式 = 职责越界,应审视:

| 反模式                                          | 历史例子                                         | 正确归属                        |
| ----------------------------------------------- | ------------------------------------------------ | ------------------------------- |
| 系统用 regex 判断 LLM "意图"(非危险信号,仅纠偏) | `WaitAndAct` / `HallucinatedConfirm` (v3.7.1 删) | 不做 — 信任 LLM                 |
| 系统强制 LLM 用特定语法格式(streaming 便利)     | "type 必须 first key" (v3.7.1 删)                | 系统 parser 容忍各种顺序        |
| 系统派生大量 LLM 输出衍生字段供后续判断         | `hasDone/hasNeedInput/blocks/...` (v3.7.1 删)    | LLM 自己负责,系统不缓存         |
| Prompt 教模型严格 schema 然后不强制             | v3.6 Rule 13 (v3.7 退化)                         | 退化为 optional,或代码强制      |
| 系统判断"该不该终止"+ 强制纠正                  | `no-marker-streak + forced-closure` (v3.7 删)    | "无 tool = 自然 final",系统不判 |
| Detector 注入 hint 教 LLM "你刚才错了"          | `PENDING_MARKER_HINT` (v3.7 删)                  | 删                              |

**正模式表** (LLM 主、系统兜底的清晰协作):

| 正模式                          | 例子                                        |
| ------------------------------- | ------------------------------------------- |
| LLM 声明意图,系统记录执行       | `pending_confirm` block + RiskGate ledger   |
| 系统遇错给结构化反馈,LLM 自适应 | `ToolErrorEnvelope` → LLM 读 hint 换策略    |
| 系统用真相数据校对 LLM 引用     | `verifyQuotedFacts` (有 tool output 可对照) |
| 系统监控资源边界,LLM 不感知     | context overflow / memory 压缩              |
| 系统计数硬阻断兜底死循环        | signature-dead-loop counter                 |
| UI 推断仅服务渲染,不回馈 loop   | `inferIntent` → `MessageBubble`             |

## 6. 四项 Cleanup 详细规格

### Cleanup 1 — 删除 `WaitAndAct` / `HallucinatedConfirm` detector

**问题**: §5 反模式表第 1 行 — 系统用 regex 做语义判断,不强制,两边不靠岸。

**安全性分析**:

- `WaitAndAct` 的核心场景:"模型说等用户但又调 side-effect tool" → `RiskGate` 仍会拦该 tool 调用(pending_confirm 主路径或 fallback regex),用户最终可 deny。**安全不丢**。
- `HallucinatedConfirm` 的核心场景:"模型说用户已确认但实际没" → `RiskGate` 不读模型自述的文本,只读真实 memory + confirm event,**结构上无法被欺骗**。Detector 本来就是多余。

**改动**:

- 删 `src/main/loop/detectors/wait-and-act-conflict.{ts,test.ts}`
- 删 `src/main/loop/detectors/hallucinated-confirm.{ts,test.ts}`
- `react-loop.ts` 的 detector 数组移除二者

### Cleanup 2 — `OutcomeFacts` 瘦身

**问题**: §5 反模式表第 3 行 — 系统为 LLM 输出派生大量衍生字段,但 Cleanup 1 后所有 talor block 衍生字段无消费者。

**字段决定矩阵**:

| 字段                                                                   | 消费方(Cleanup 1 后)                          | 处理   |
| ---------------------------------------------------------------------- | --------------------------------------------- | ------ |
| `hasToolCall`                                                          | react-loop 主循环 + detector                  | 保留   |
| `hasText`                                                              | 多个 detector                                 | 保留   |
| `allToolsFailed`                                                       | signature-dead-loop + failure-streak          | 保留   |
| `isSubagentFailure`                                                    | failure-streak                                | 保留   |
| `signature`                                                            | signature-dead-loop                           | 保留   |
| `toolNames`                                                            | 仅 WaitAndAct 用 → 删后无                     | **删** |
| `hasMarker / hasTermination / hasLegacyMarker`                         | 死                                            | **删** |
| `hasDone / hasNeedInput / hasBlocked / hasPendingConfirm / hasWarning` | 仅 WaitAndAct/HallucinatedConfirm 用 → 删后无 | **删** |
| `blocks / invalidBlocks`                                               | 死                                            | **删** |

**改动**:

- `outcome-facts.ts`: `classify()` 不再调 `parseTalorBlocks`(切断 main → shared/talor-blocks 运行时依赖),只派生维度 A 信号
- `OutcomeFacts` interface 从 16 字段瘦到 5 字段
- 5 个 detector test fixture 删对应初始化字段

### Cleanup 3 — 删 "type 必须 first key" 约束 + streaming detector 位置无关

**问题**: §5 反模式表第 2 行 — 系统侧 streaming 便利压给 LLM。

**当前 regex**:

````ts
const STREAMING_TYPE_RE = /```talor[ \t]*\n\s*\{\s*"type"\s*:\s*"(\w+)"/g
````

要求 type 紧跟 `{`。

**改进**:

````ts
const STREAMING_TYPE_RE = /```talor[ \t]*\n[\s\S]*?"type"\s*:\s*"(\w+)"/g
````

允许 type 在 JSON 任意位置。

**边界 case**: 模型在 JSON value 里写假 `"type":"..."` → streaming 期间可能短暂误判类型,但 JSONC 闭合后 parser 取真实 block 仍正确。实际场景罕见,代价极小。

**改动**:

- `talor-block-parser.ts`: regex 改位置无关
- `talor-block-parser.test.ts`: 加 "type 不在 first 位仍能识别" 回归 case
- `SystemPlugin.ts` Rule 13 删 `The \`type\` field must be the FIRST key (streaming detection).` 一行
- `SystemPlugin.test.ts` 移除 `'FIRST key'` 字符串断言

### Cleanup 4 — `intent-classifier` 移位 + 同迁辅助函数

**问题**: §5 正模式表 "UI 推断仅服务渲染" — 位置应明示职责。

**目标结构**:

```
src/shared/
├── talor-blocks/                     ← 协议本体
│   ├── talor-block-schema.ts
│   ├── talor-block-parser.ts
│   └── talor-block-parser.test.ts
└── ui-rendering/                     ← UI 工具
    ├── intent-classifier.ts          ← 从 talor-blocks/ 搬
    ├── intent-classifier.test.ts
    ├── text-heuristics.ts             ← 从 outcome-facts.ts 拆出
    └── text-heuristics.test.ts
```

`text-heuristics.ts` 接收: `hasTerminationMarker` / `hasTerminationInText` / `looksLikeOpenQuestion` / `TERMINATION_MARKERS`。

**改动**:

- `git mv` intent-classifier 两文件
- 新建 `text-heuristics.ts` + 测试,从 outcome-facts 切出
- `outcome-facts.ts` 删上述 helper (它们不再服务 detector,只服务 UI)
- 改 `MessageBubble.tsx` import 路径
- intent-classifier 内部可 import `./text-heuristics` 复用(实际上 intent-classifier 自己有 regex,不依赖 helper)

## 9. 不做(YAGNI / 留 v3.8+)

- WaitAndAct 升级为硬阻断 — regex 准确率不够,会误阻断合理操作
- fallback regex 改 LLM 调用 — 成本太高且 fallback 触发频率本就低
- Rule 11/12 改 inferIntent UI 提示 — 软规则现状可接受
- talor block 协议完全删 — 强模型仍能用,UI 卡片体验有价值
