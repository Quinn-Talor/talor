## 1. 背景

### 1.1 v3.6 投入回顾

[v3.6 方案](2026-05-12-talor-block-protocol.md) 定义了 5 层架构(L0 协议 / L1 流程 / L2 语义 / L3 风险 / L4 审计 / L5 UI),8 个 commit 共改 ~2700 行,新增 71+ 测试。

### 1.2 落地后的真实表现

截图回归(2026-05-12 实测):

- Step 1: 模型列 3 个问题 + "你先答第一个" → 无 talor marker → no-marker streak +1
- Step 2: 用户回 "保持节奏" → 模型重问 → no-marker streak +2
- Step 3: 模型再问 → no-marker streak +3 → **forced-closure 启动**
- Step 4: forced-closure 下,模型凭空自答 "好,日本市场🇯🇵" + 又问一堆问题
- 末尾: postProcess 补 ⏸ Blocked

**6 个补丁 commit(ea895d4 / 412d027 / 7d368dd / 4c3bc99 等)没修到根因**。每修一处就发现下一个边界。

### 1.3 根因诊断

**错误假设**: "模型一步无 tool + 无 marker = 想停但没收尾,需要代码强制纠正"

**实际**: 现代 LLM 自然语言能力足够,无 marker 也可以是合法终止:

- 模型答完问题、没下一步 → 自然结束
- 模型问用户、等回答 → 自然结束(问号本身就是信号)
- 模型说"无法做" → 自然结束

把"无 marker"当 bug 引入 `no-marker-streak` + `forced-closure` 是**过度工程**。`forced-closure` 在压力下让模型自答,制造比解决更多的灾难。

## 2. 设计原则

继承 v3.6 的 6 条 + 新增 4 条:

| #      | 原则                               | 实质                                                      |
| ------ | ---------------------------------- | --------------------------------------------------------- |
| 1-6    | (v3.6 原有)                        | 分层 / 扩展 / 保守 / 零回归 / 量化 / 判断给 LLM           |
| **7**  | **容错两个维度独立处理**           | A(代码错)=代码强制,B(意图)=信任 LLM                       |
| **8**  | **不强制纠正 LLM 自然语言**        | LLM 不发 marker 不是 bug。强制纠正的代价(自答灾难) > 收益 |
| **9**  | **代码只在"安全 / 审计"上强制**    | pending_confirm + RiskGate + Ledger 必须代码强制          |
| **10** | **talor block 退化为可选 UI 增强** | 强模型 + 高级 UX 用;弱模型不发也能正常工作                |

## 3. 三阶段实施方案

### Phase A — 砍掉强制纠正(本次实施重点)

删除 `no-marker-streak` / `forced-closure` 整条路径,react-loop 退化为"无 tool = 终止"。

**Done 标准**:

- 现有测试通过(预计 -40,涉及 forced-closure / no-marker-streak)
- 截图同类场景走自然 final 路径,不再 forced-closure
- 代码 -500 行,prompt -300 token

### Phase B — inferIntent 仅作 UI 辅助

`looksLikeOpenQuestion` 升级为多信号 voting,**仅决定 UI 卡片渲染**,不影响 loop 控制。

**Done 标准**:

- 强模型 emit talor block → 渲染对应卡片(主路径)
- 弱模型不发 block + 文本含问号 → UI 推断为 need_input 卡片(启发式路径)
- inferIntent 对截图回归文本召回 ≥80%

### Phase C — 远期评估(待定)

`pending_confirm` 是否改为 framework tool 化、talor block 协议是否完全砍掉,看 Phase A/B 落地后数据再定。

### 4.2 修改清单

#### 4.2.1 `runReactStep` 简化逻辑

**之前**: 无 tool + 无 marker → intermediate + continue + 进 no-marker streak

**之后**(极简): 无 tool + 任何 text → 自然 final(用 messageId persist,`exitReason='no_tool_calls'`)

```ts
if (stepToolCalls.length === 0) {
  if (stepText) {
    // 任何无 tool 的 text 输出 → 自然 final
    persistFinal(stepText)
    return { shouldContinue: false, exitReason: 'no_tool_calls', ... }
  }
  // 空 → empty_text(既有路径)
  return { shouldContinue: false, exitReason: 'empty_text', ... }
}
// 有 tool → 落 assistant+tool batch + continue(不动)
```

#### 4.2.4 Prompt 简化

**Rule 13 之后**(简化版 ~30 行):

````
13. (Optional) Mark turn-ending decisions with talor blocks for richer UI.

    The framework infers your intent from natural text. Optionally,
    you can emit a structured talor block as the last block of your reply
    for the framework to render a typed UI card (done / need_input / blocked).

      ```talor
      {"type":"done","summary":"..."}
````

## 5. Phase B 详细规格

### 5.1 inferIntent 多信号 voting

新文件 `src/main/loop/intent-classifier.ts`:

```ts
export interface IntentInference {
  type: 'done' | 'need_input' | 'blocked' | null
  confidence: number
  signals: string[]
}

const SIGNALS = {
  done: [
    {
      re: /(?:已(?:完成|成功|搞定|插入|更新|删除|创建)|completed?|finished|all set|ready)\b/i,
      weight: 2,
    },
    { re: /(?:以下是|总结|here'?s|in summary)/i, weight: 1 },
    { re: /[?？]/, weight: -1.5 }, // 反信号
  ],
  need_input: [
    { re: /[?？]/, weight: 2 },
    { re: /\S+\s+\/\s+\S+\s+\/\s+\S+/, weight: 1.5 }, // X / Y / Z
    {
      re: /(?:请(?:告诉|告知|确认|选|提供)|等(?:你|您)|let me know|tell me|which (?:one|do|would))/i,
      weight: 1.5,
    },
  ],
  blocked: [
    {
      re: /(?:找不到|不存在|缺(?:少|失)|无法|cannot|unable to|failed to|missing|not found)/i,
      weight: 2,
    },
    { re: /(?:error|exception|refused|denied|timeout|ENOENT|ECONN)/i, weight: 1 },
  ],
}

export function inferIntent(text: string): IntentInference
```

阈值 1.5,低于此则 `type: null`(让 UI 走普通 bubble)。

### 5.2 UI 渲染映射

```ts
// MessageBubble — assistant message 渲染
const blocks = parseTalorBlocks(textContent).blocks
if (blocks.length > 0) {
  renderTalorCards(blocks) // 主路径,显式 > 推断
} else if (isAssistantFinal) {
  const inferred = inferIntent(textContent)
  if (inferred.type) {
    renderInferredCard(inferred, textContent) // 带 "⚙️ inferred" 徽章
  } else {
    renderPlainBubble(textContent)
  }
}
```

### 5.3 设计要点

1. **inferIntent 完全是 UI 层判断**,不进入 react-loop 控制流(对比 v3.6 的 `looksLikeOpenQuestion` 影响 loop 终止判定)
2. **强模型主动 emit talor block**:UI 优先按 block 渲染(显式 > 推断)
3. **弱模型不发 block**:UI 推断后渲染,标"⚙️"暗示是推断
4. **threshold 兜底**:推断置信度低于 1.5 → 退化为普通 MessageBubble

## 10. FAQ / 取舍说明

### 10.1 为什么不彻底删 talor block 协议?

强模型(Claude / GPT-4)在适当 prompt 下能稳定 emit `\`\`\`talor done\`\`\``,UI 卡片体验明显优于推断。**保留 talor block 是为了不浪费强模型的能力**,但**不依赖**它工作。

### 10.2 为什么保留 RiskGate / pending_confirm?

副作用(SQL 写 / 文件改 / 外部 API)的授权是 critical safety,**必须代码强制**。模型说"我已得到用户批准"绝不能直接信。

### 10.3 没有 forced-closure,模型真的会自然结束吗?

会。现代 LLM 训练数据里大量"完成任务 → 答复用户 → 不再调工具"的范例。`no-marker-streak` 是在过度补偿一个**不存在的问题**。截图里模型连续 3 步纯文本不停,实际原因是 prompt 教它"必须 emit marker",反而让它困惑。

### 10.4 为什么不直接做 tool 化 closure?

用户排除(认为多一步 LLM 推理浪费 token,虽然实际 zero-cost)。**尊重决策**,Phase A 仍以"信任 LLM 自然语言"路径推进。
