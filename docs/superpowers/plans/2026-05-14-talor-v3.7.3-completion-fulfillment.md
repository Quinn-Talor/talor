## 0. TL;DR

v3.7.3 一并解决两件事:

**事 1 — promise-then-stop 中断**:Principle 12 ("Promise then call") 是 Talor 唯一**没有协议-行为配对**的硬规则。弱模型说"现在写入文档:"然后停手,框架按 v3.7 默认 natural FINAL 收尾,任务中断。

补 `pending_continuation` block (talor 协议)+ `TurnEndPolicy` Strategy+Chain 抽象,形成完整 rule-block-policy 三元组。

**事 2 — SDK 信号未被系统化使用**:react-loop 的 step-end 决策**基于 `stepToolCalls.length + stepText` 二元启发**,完全忽略 SDK 已经给的 `finishReason`/`usage`/`providerMetadata` 等结构化信号。这些信号按 J-SHOULD-2 应该是"LLM 自陈"和"运行时真相"的**一等公民**。

把 SDK 信号按协作模型分类纳入,react-loop step-end 重构为 `finishReason` switch,JudgePolicy 改用 `generateObject` 替代字符串解析,token 估算改用 `usage` 精确值。

**完全 J-SHOULD-2 合规** — 零 regex 介入意图判断,LLM 主声明 + SDK 结构化信号 + LLM judge 兜底 + system observation detector 四层防御纵深。

## 1. 问题溯源

### 1.1 事故现场

```
2026-05-14 dev session, agent: __chat__, provider: deepseek2

step 5-8:  并行 mysql_query × 40 (SHOW CREATE TABLE × 40) — 全部成功
step 9:    messages=21, stream 50.13s, 输出 37 chars text, 零 tool call
           → react-loop 判定 "natural FINAL" break

37 字符文本: "全部 40 张表的数据都获取完成了。现在写入本地 Markdown 文档:"
```

LLM 承诺"现在写入"后没 emit `write` tool call。

### 1.2 当前 8 条 loop 终止路径全是"观察信号"

| #   | exitReason         | 触发                         | 判断依据                                    |
| --- | ------------------ | ---------------------------- | ------------------------------------------- |
| 1   | `no_tool_calls`    | step 末有文本 + 无 tool call | 观察:`stepToolCalls.length===0 && stepText` |
| 2   | `empty_text`       | step 末无文本 + 无 tool call | 观察:同上                                   |
| 3   | `abort`            | AbortSignal 触发             | 观察:信号                                   |
| 4   | `max_steps`        | step ≥ 1000                  | 观察:计数                                   |
| 5   | `fallback_summary` | 整轮零文本 → 强制 summary    | 观察:计数                                   |
| 6   | `repeated_error`   | signature 重复 OR N 连续失败 | 观察:hash/计数                              |
| 7   | `tool_only_loop`   | N 步 tool-only 零文本        | 观察:计数                                   |
| 8   | `context_overflow` | tokens ≥ 100%                | 观察:**估算的** token 数                    |

**两个结构性缺陷**:

**缺陷 A**:8 条路径全是**系统侧观察 (counts / hash / 信号量)**,没有任何一条问"LLM 是否完成了它说要做的事"。
**缺陷 B**:**完全忽略 SDK 已经给的 LLM 自陈信号** — `finishReason: 'stop' / 'length' / 'tool-calls'` 同时存在,react-loop 却用 `stepToolCalls.length + stepText` 自己推断。

### 1.3 Rule-Block-Policy 配对矩阵 (改前)

```
Principle 14  ────  pending_confirm        ────  RiskGate policy        ✅ 完整
Principle 9   ────  ─                       ────  runForcedSummary       ✅ 兜底
Principle 11  ────  ─                       ────  UI inferIntent         ✅ UI 兜底
Principle 13  ────  done/need_input/blocked ────  仅 UI 卡片渲染         ⚠️ loop 不消费
Principle 12  ────  ⚠️ 没有 block            ────  ⚠️ 没有 policy         ❌ 协议-行为缺口
```

## 2. 设计原则

### 2.1 SDK 信号按 J-SHOULD-2 分类

AI SDK v6 给我们的信号按"谁说的、说什么"分两类,完美映射到 J-SHOULD-2 协作矩阵:

```
┌────────────────────────────────────────────────────────────────────┐
│  类别 A: LLM 自陈信号 (LLM 通过 SDK 告诉系统它的意图/状态)         │
├────────────────────────────────────────────────────────────────────┤
│  • finishReason: 'stop' / 'length' / 'tool-calls' / 'content-      │
│      filter' / 'error' / 'other' / 'unknown'                       │
│  • content parts: text / reasoning / tool-call                      │
│  • text 内的 talor block (LLM 主动 emit 的结构化决策声明)           │
├────────────────────────────────────────────────────────────────────┤
│  → 按 J-SHOULD-2 处理方式: **信任 LLM,policy 直接消费**             │
└────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────┐
│  类别 B: 运行时真相 (provider/SDK 测得的客观数据,非 LLM 主张)      │
├────────────────────────────────────────────────────────────────────┤
│  • usage: { inputTokens, outputTokens, totalTokens }                │
│  • durationMs                                                       │
│  • providerMetadata: { anthropic.cacheReadInputTokens, ... }        │
│  • warnings: provider 警告 (e.g. 不支持的参数)                      │
├────────────────────────────────────────────────────────────────────┤
│  → 按 J-SHOULD-2 处理方式: **系统兜底校验,detector / Ledger 消费**  │
└────────────────────────────────────────────────────────────────────┘
```

**关键洞察**:`finishReason` 本质上是**第二个 LLM 自陈通道**,与 talor block 平行 — talor block 在 LLM 的 text content 里,`finishReason` 在 SDK metadata 里,**两者都是 LLM 主声明**,都应该被 policy 链优先消费。

### 2.2 判断主体严格归属

| 决策点                         | 判断主体                 | 用什么判                                   |
| ------------------------------ | ------------------------ | ------------------------------------------ |
| LLM 是否声明了 turn-end 意图   | 系统                     | SDK `finishReason` + parser (确定性)       |
| LLM 是否声明了"继续"           | 系统                     | parseTalorBlocks (确定性)                  |
| LLM 没声明时是否要续做         | **LLM judge (二审 LLM)** | LLM 语义推理 + `generateObject` Zod 强类型 |
| Token 是否超限                 | 系统                     | SDK `usage` (精确,不再估算)                |
| 系统能不能用 regex 判 LLM 意图 | **永不**                 | —                                          |

### 2.3 完整配对 (改后)

```
Principle 12  ────  pending_continuation     ────  PendingContinuationBlockPolicy
  ("promise then call")  block (talor 协议新增)        Strategy+Chain 链
                                              ────  finishReason='length' → 自动 continue
                                                    (SDK 信号免费给的)
                                              ────  JudgeCompletionPolicy (兜底)
                                                    LLM-as-judge 二审 (generateObject)
```

## 3. 协议扩展 — `pending_continuation` block (最简版)

### 3.1 Schema

```ts
// src/shared/talor-blocks/talor-block-schema.ts (新增)

/**
 * 续做声明 — LLM 在本 turn 完成了一部分但承诺下一步还有动作,框架据此续做 loop。
 *
 * 设计:零必填字段。`type` 本身就是全部语义信号。
 *
 * 为什么不要 next_action 字段:
 *   - LLM emit 此 block 之前的 text 已经说了"现在写入文档:"——这是单一事实源
 *   - 强制 next_action 是字面重复,LLM 配合阻力↑,paraphrase 漂移风险↑
 *   - 续做 hint 直接引用前文 ("look back at your last response"),不需 LLM 二次表达
 *
 * 与 Principle 12 "Promise then call" 配对,四种 turn-end 形态:
 *   A. 执行 — 同 turn 调 tool (不需要本 block)
 *   B. 延后 — emit pending_continuation,系统续做
 *   C. 结束 — emit done/need_input/blocked
 *   D. ❌ 说要做但没动手也没 block — JudgePolicy 二审兜底
 */
export interface PendingContinuationBlock {
  type: 'pending_continuation'
  /** 选填: 为什么 turn 在这里断 (UI 显示 + 日志 reviewer 用,框架不消费) */
  reason?: string
}
```

### 3.2 Parser/Validator 改动

| 文件                                          | 改动                                                                   |
| --------------------------------------------- | ---------------------------------------------------------------------- |
| `talor-block-schema.ts`                       | `TalorBlock` union 加 `PendingContinuationBlock`,`V1_BLOCK_TYPES` 加项 |
| `talor-block-parser.ts` `isV1Type`            | 加 `'pending_continuation'`                                            |
| `talor-block-parser.ts` `validateBlockFields` | 加 case:`type === 'pending_continuation'` 永远 valid (无必填字段)      |

### 3.3 UI 渲染

`TalorBlockRenderer.tsx` 加渲染 case:

```tsx
case 'pending_continuation':
  return <PendingContinuationCard reason={block.reason} />

// 卡片视觉: 浅色 chip + ⏵ 图标 + 选填 reason
// "▶ 续做中..." (无 reason) 或 "▶ 续做中:数据已收集,准备写入" (有 reason)
// 表达"任务未结束,框架正在驱动续做"
```

## 4. 架构 — `TurnEndPolicy` Strategy + Chain

### 4.1 接口

```ts
// src/main/loop/turn-end-policies/types.ts (新增)

export interface TurnEndDecision {
  action: 'final' | 'continue' | 'no-opinion'
  exitReason?: LoopExitReason
  injectHint?: string
  reason: string // 日志/审计
}

export interface PolicyContext {
  agent: import('../../agent/agent').Agent
  sessionId: string
  stepIndex: number
  abortSignal: AbortSignal
  /** SDK 给的 LLM 自陈信号 (一等公民) */
  sdkSignals: {
    finishReason: import('ai').FinishReason
    usage?: { inputTokens?: number; outputTokens?: number; totalTokens?: number }
    providerMetadata?: Record<string, unknown>
    warnings?: import('ai').LanguageModelV2CallWarning[]
  }
  /** Judge call 用的 model 工厂 (JudgePolicy 才需要) */
  judgeProvider?: () => Promise<import('ai').LanguageModel>
}

export interface TurnEndPolicy {
  readonly name: string
  evaluate(outcome: StepOutcome, ctx: PolicyContext): Promise<TurnEndDecision>
}
```

**关键**:`PolicyContext` 把 SDK 信号作为**一等参数**传入,所有 policy 都能直接基于 LLM 自陈 + 运行时真相决策。

### 4.2 五个 Policy 实现 (按链顺序)

```ts
// turn-end-policies/sdk-finish-reason.ts (P0 — SDK 信号最优先)
export class SdkFinishReasonPolicy implements TurnEndPolicy {
  name = 'sdk-finish-reason'
  async evaluate(_outcome, ctx) {
    const reason = ctx.sdkSignals.finishReason
    switch (reason) {
      case 'tool-calls':
        // 不该走到这里 — react-loop 在 stepToolCalls.length>0 时直接续 loop
        return { action: 'no-opinion', reason: 'has tool calls, should not reach turn-end policy' }

      case 'length':
        // SDK 直接告诉我们 max_tokens 触发,模型没说完
        return {
          action: 'continue',
          exitReason: 'truncated',
          injectHint:
            `Your previous response was truncated mid-output by the provider's ` +
            `max_tokens limit. Continue from where you left off; do NOT repeat ` +
            `already-produced content. If the truncated content was a single ` +
            `large artifact (file write, long markdown), consider splitting into ` +
            `smaller chunks.`,
          reason: `finishReason='length' (max_tokens hit)`,
        }

      case 'content-filter':
        return {
          action: 'final',
          exitReason: 'content_filter',
          reason: 'provider content filter triggered',
        }

      case 'error':
      case 'other':
      case 'unknown':
        // 异常但非截断,fail-open 给下一层
        return { action: 'no-opinion', reason: `finishReason='${reason}', falling through` }

      case 'stop':
      default:
        // 模型主声明自然停 — 不代表它没承诺更多,需要继续看 block / judge
        return { action: 'no-opinion', reason: 'stop, falling through to block/judge layers' }
    }
  }
}

// turn-end-policies/explicit-termination.ts (P1)
export class ExplicitTerminationBlockPolicy implements TurnEndPolicy {
  name = 'explicit-termination'
  async evaluate(outcome) {
    const { blocks } = parseTalorBlocks(outcome.stepText)
    const terminal = blocks.find(
      (b) => b.type === 'done' || b.type === 'need_input' || b.type === 'blocked',
    )
    if (!terminal) return { action: 'no-opinion', reason: 'no terminal block' }
    return {
      action: 'final',
      exitReason: 'declared_final',
      reason: `LLM emitted ${terminal.type} block`,
    }
  }
}

// turn-end-policies/pending-continuation.ts (P2)
export class PendingContinuationBlockPolicy implements TurnEndPolicy {
  name = 'pending-continuation'
  async evaluate(outcome) {
    const { blocks } = parseTalorBlocks(outcome.stepText)
    const hasContinuation = blocks.some((b) => b.type === 'pending_continuation')
    if (!hasContinuation) return { action: 'no-opinion', reason: 'no pending_continuation' }
    return {
      action: 'continue',
      exitReason: 'continuation_injected',
      injectHint:
        `Continuation reminder: your previous step ended with a ` +
        `pending_continuation block but no tool call. Look back at what you ` +
        `said in that step and execute the action you committed to. If you've ` +
        `changed your mind, emit done/blocked instead.`,
      reason: 'LLM declared pending_continuation',
    }
  }
}

// turn-end-policies/judge.ts (P3 — 兜底)
import { generateObject } from 'ai'
import { z } from 'zod'

const JudgeSchema = z.object({
  decision: z.enum(['COMPLETE', 'CONTINUE']),
  reason: z.string().min(1).max(200),
})

export class JudgeCompletionPolicy implements TurnEndPolicy {
  name = 'judge'
  constructor(
    private cfg: {
      enabled: boolean
      judgeModel: () => Promise<LanguageModel>
      timeoutMs?: number // default 5000
    },
  ) {}

  async evaluate(outcome, ctx) {
    if (!this.cfg.enabled) return { action: 'no-opinion', reason: 'disabled' }
    try {
      const { object } = await Promise.race([
        generateObject({
          model: await this.cfg.judgeModel(),
          schema: JudgeSchema,
          prompt: buildJudgePrompt(outcome.stepText),
          abortSignal: ctx.abortSignal,
        }),
        timeoutAfter(this.cfg.timeoutMs ?? 5000),
      ])
      if (object.decision === 'COMPLETE') {
        return {
          action: 'final',
          exitReason: 'judge_complete',
          reason: `judge says complete: ${object.reason}`,
        }
      }
      return {
        action: 'continue',
        exitReason: 'continuation_injected',
        injectHint:
          `Continuation reminder (auto-detected): ${object.reason}\n` +
          `Your previous output appeared to promise an action without ` +
          `executing it. Execute the action now, or emit done/blocked/` +
          `need_input if you've changed your mind.`,
        reason: `judge says continue: ${object.reason}`,
      }
    } catch (err) {
      // fail-open: ZodError / 超时 / provider 错误 → 退到下个 policy
      log.warn('[JudgeCompletionPolicy] judge failed, fail-open:', err)
      return {
        action: 'no-opinion',
        reason: `judge failed: ${err instanceof Error ? err.message : String(err)}`,
      }
    }
  }
}

// turn-end-policies/legacy.ts (P4 — 兜底,永不 no-opinion)
export class LegacyNaturalFinalPolicy implements TurnEndPolicy {
  name = 'legacy-natural-final'
  async evaluate() {
    return {
      action: 'final',
      exitReason: 'no_tool_calls',
      reason: 'v3.7 natural FINAL fallback',
    }
  }
}
```

### 4.3 默认链组装

```ts
// turn-end-policies/default-chain.ts
export function buildDefaultChain(opts: {
  judgeEnabled: boolean
  judgeModel: () => Promise<LanguageModel>
}): TurnEndPolicy[] {
  return [
    new SdkFinishReasonPolicy(), // P0: SDK 信号
    new ExplicitTerminationBlockPolicy(), // P1: 显式终止 block
    new PendingContinuationBlockPolicy(), // P2: 续做 block
    new JudgeCompletionPolicy({
      // P3: LLM judge 兜底
      enabled: opts.judgeEnabled,
      judgeModel: opts.judgeModel,
    }),
    new LegacyNaturalFinalPolicy(), // P4: v3.7 兜底
  ]
}
```

**链顺序逻辑**:

- P0:**SDK 给的 finishReason 是 provider 测得的硬信号**,优先级最高(`length` 截断、`content-filter` 拒绝是 SDK 已经告诉我们的事实,无需再判)
- P1-P2:**LLM 主声明 block**(自陈 / 协议级)
- P3:**LLM 二审兜底**(LLM 没声明时的语义判断)
- P4:**v3.7 兼容兜底**(judge 失败 / 关闭时)

### 4.4 react-loop step-end 重构 — 基于 `finishReason`

```ts
// 旧实现 (启发式):
if (stepToolCalls.length === 0) {
  if (stepText) return { ...natural FINAL... }
  else        return { ...empty_text... }
}

// 新实现 (SDK 信号驱动):
const finishReason = await result.finishReason
const usage = await result.usage
const providerMetadata = await result.providerMetadata
const warnings = await result.warnings

// 工具路径: SDK 明确说有 tool-calls,继续 loop
if (finishReason === 'tool-calls' || stepToolCalls.length > 0) {
  return { ...persist assistant+tool, shouldContinue: true... }
}

// 空文本路径: 维持原状
if (!stepText) {
  return { ...empty_text..., exitReason: 'empty_text' }
}

// 候选 FINAL 路径: 跑 turn-end policy 链
const decision = await runPolicyChain(outcome, {
  agent, sessionId, stepIndex, abortSignal,
  sdkSignals: { finishReason, usage, providerMetadata, warnings },
  judgeProvider,
}, ctx.turnEndPolicies)

if (decision.action === 'final') {
  return { ...persist assistant text..., exitReason: decision.exitReason }
}

// 'continue': 注入 hint + 续 loop
return {
  stepText, wroteAssistantFinal: false, shouldContinue: true,
  injectedHint: decision.injectHint,
  exitReason: decision.exitReason,
  durationMs, toolNames: [], signature: '',
  allToolsFailed: null, containsSubagentFailure: false,
}
```

## 5. SDK 信号在其他位置的系统化使用

不只是 turn-end policy 用,把 SDK 信号纳入 J-SHOULD-2 一等公民意味着 react-loop **所有需要"模型停了吗 / 用了多少 token / 命中 cache 吗"信息的地方都该用 SDK 真相值,不再估算**。

### 5.1 Context 预算检测 — 用 `usage` 精确值替代 `estimate()`

**当前实现**(react-loop.ts:223-273):

```ts
let estimatedTokens = 0
for (const m of messages) {
  const text = typeof m.content === 'string' ? m.content : JSON.stringify(m.content)
  estimatedTokens += estimate(text)  // ← 启发式估算 (字符数 / 3-ish)
}
const usageRatio = estimatedTokens / limit
if (usageRatio >= 1.0) { ...auto-halt... }
```

`estimate()` 在 `memory/types.ts` 是粗糙启发(`Math.ceil(text.length / 3)`)。

**改后**:首次 step 用估算(因为还没真实 usage),从第二步起用上一步 `result.usage.inputTokens`(provider 算的精确值):

```ts
// react-loop state
let lastUsage: { inputTokens?: number; totalTokens?: number } | undefined

// step start
const estimatedTokens = lastUsage?.inputTokens
  ?? messages.reduce((s, m) => s + estimate(...), 0)  // 第一 step fallback

// ...streamText 跑完后...
lastUsage = await result.usage  // 更新供下一步用
```

**好处**:

- 估算偏差 (典型 ±15%) 消除 → context_overflow halt 阈值更准
- 接近 100% 时不再误判 → 减少误 halt
- 副产物:Ledger / 日志可记真实 token 用量,后续做 cost-aware policy 有数据

按 J-SHOULD-2 分类:`usage` 属于**运行时真相信号** → 系统消费 ✅

### 5.2 Provider 警告纳入 Ledger — `warnings` 字段

SDK `result.warnings` 会暴露 provider 警告 (例如 OpenAI "tool_choice ignored"、Anthropic "cacheControl unsupported on this model")。今天 react-loop 完全忽略。

**改动**:react-loop step 末 push 到 SideEffectLedger 一条 `warning` 类型记录,debug 用:

```ts
const warnings = await result.warnings
if (warnings && warnings.length > 0) {
  for (const w of warnings) {
    log.warn(`[ReactLoop] provider warning: ${w.message}`)
    sideEffectLedger.record({
      ...,
      op: 'provider:warning',
      target: w.type,
      preview: w.message,
      confirmed_by: 'system',
      user_decision: 'auto',
    })
  }
}
```

后续 review session 时 grep ledger 能快速发现 provider 兼容问题。

### 5.3 Cache hit 可观测 — `providerMetadata`

Anthropic 在 `result.providerMetadata.anthropic` 暴露:

- `cacheCreationInputTokens` — 创建 cache 的 tokens
- `cacheReadInputTokens` — 命中 cache 的 tokens (大量节省)

把这些纳入 ledger / 日志,后续做 prompt caching(v3.7.4 单独 plan)时可以监测命中率。

```ts
const meta = await result.providerMetadata
const cacheRead = meta?.anthropic?.cacheReadInputTokens as number | undefined
const cacheCreate = meta?.anthropic?.cacheCreationInputTokens as number | undefined
if (cacheRead || cacheCreate) {
  log.info(`[ReactLoop] cache: read=${cacheRead} create=${cacheCreate}`)
}
```

## 6. 防滥用 — `ContinuationChainDetector`

防止 LLM 连续 emit pending_continuation 而不真动手:

```ts
// loop/detectors/continuation-chain.ts
export class ContinuationChainDetector implements LoopDetector {
  readonly name = 'continuation-chain'
  private chain = 0

  observe(facts, _stepIndex, raw) {
    if (!raw) return NO_TRIGGER
    const hasContinuation = parseTalorBlocks(raw.stepText).blocks.some(
      (b) => b.type === 'pending_continuation',
    )

    if (hasContinuation && facts.toolNames.length === 0) {
      this.chain++
      if (this.chain >= 3) {
        log.warn(`[ContinuationChainDetector] chain reached ${this.chain}, breaking`)
        return {
          triggered: true,
          exitReason: 'continuation_chain',
          markFinal: true,
        }
      }
    } else {
      this.chain = 0
    }
    return NO_TRIGGER
  }

  nextHint(): string | null {
    if (this.chain === 2) {
      return (
        `You've emitted pending_continuation twice consecutively without ` +
        `executing the promised tool call. The next pending_continuation ` +
        `will terminate the turn. Execute the action now, or emit ` +
        `blocked/done if you cannot proceed.`
      )
    }
    return null
  }
}
```

注册到 react-loop detector 链(append 在现有 3 个之后)。

```text
### J-SHOULD-3 · AI SDK 信号按"LLM 自陈 / 运行时真相"两类消费

**类别 A — LLM 自陈** (LLM 通过 SDK 告诉系统):

```

## 12. 测试矩阵

### 12.1 协议层

- ✅ `pending_continuation` 合法解析
- ✅ `pending_continuation` 无 reason (最简形式) 通过
- ✅ `pending_continuation` 含 reason 解析正确
- ✅ JSONC 容错

### 12.2 Policy 层

**SdkFinishReasonPolicy**:

- ✅ `finishReason='length'` → continue + injectHint TRUNCATION
- ✅ `finishReason='content-filter'` → final + exitReason='content_filter'
- ✅ `finishReason='stop'` → no-opinion
- ✅ `finishReason='tool-calls'` → no-opinion (此路径不该到 turn-end policy)
- ✅ `finishReason='error'/'other'/'unknown'` → no-opinion

**ExplicitTerminationBlockPolicy**:

- ✅ done block → final
- ✅ need_input → final
- ✅ blocked → final
- ✅ 无 → no-opinion
- ✅ done + pending_continuation 共存 → P1 优先,returns done final

**PendingContinuationBlockPolicy**:

- ✅ 含 block (无 reason) → continue + 通用 hint
- ✅ 含 block (有 reason) → continue + hint
- ✅ 无 → no-opinion

**JudgeCompletionPolicy**:

- ✅ disabled → no-opinion
- ✅ `generateObject` 返 COMPLETE → final
- ✅ `generateObject` 返 CONTINUE → continue + injectHint
- ✅ timeout → no-opinion (fail-open)
- ✅ ZodError (schema 不符) → no-opinion (fail-open)
- ✅ provider throw → no-opinion (fail-open)
- ✅ judge 用 `generateObject` 而非字符串解析 (验证调用方式)

**LegacyNaturalFinalPolicy**:

- ✅ 永远返 final + `'no_tool_calls'`

### 12.3 Chain 层

- ✅ 链顺序:P0 (SDK) → P1 (终止 block) → P2 (continuation) → P3 (judge) → P4 (legacy)
- ✅ `finishReason='length'` 不会跑到 P1 之后 (SDK 信号优先)
- ✅ `finishReason='stop'` + done block → P1 wins
- ✅ `finishReason='stop'` + 无 block + judge disabled → P4 (legacy)
- ✅ `finishReason='stop'` + 无 block + judge throws → P4 (legacy)

### 12.4 SDK 信号集成

- ✅ `usage.inputTokens` 用于第二步起 context 预算检测 (替代 estimate)
- ✅ `warnings` 入 Ledger
- ✅ `providerMetadata.anthropic.cacheReadInputTokens` 日志记录
- ✅ `usage` 缺失 → fallback to estimate (不崩溃)
- ✅ `warnings` 缺失 → silent (不崩溃)

### 12.5 ContinuationChainDetector

- ✅ 单次 pending_continuation → 不触发
- ✅ 连续 2 次 → nextHint 警告
- ✅ 连续 3 次 → triggered
- ✅ 中间穿插 tool call / done block → reset

### 12.6 端到端 (mock LLM + SDK)

- ✅ promise-then-stop 重现 → judge CONTINUE → 下一步 LLM emit write → 完成
- ✅ `finishReason='length'` 截断 → 自动 continue + truncation hint → LLM 续输出
- ✅ 强模型主动 emit pending_continuation → 不触发 judge
- ✅ 弱模型滥用 pending_continuation × 3 → ContinuationChainDetector break
- ✅ judge disabled → 等价 v3.7 行为 (LegacyNaturalFinalPolicy)
