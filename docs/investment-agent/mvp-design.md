# A股标的投资 Agent — MVP 完整方案

> 本文是投资 agent 的**权威系统设计**。相较早期分层方案大幅收敛:**砍掉宏观/行业独立卡、版本化共享视图、跨卡传导、5维矩阵、选股发现流**(推迟 v2)。MVP 只做一件事:**让一张标的卡走完「建立 → 逻辑跟踪 → 决策 → 复盘」的完整闭环**。
>
> 方法论依据:`docs/investment-agent/README.md`(散户认知 / 三层研究 / 逻辑卡 / 证伪 / 估值 / 纪律 / 复盘)。
> 实施按 **klook-vibe** 规范(`vibe/features/investment-agent/`)。`invest_lib` = §8 的 6 个纯函数,落 `src/main/invest/compute/`。

---

## §0 设计原则(贯穿全文的铁律)

1. **任务边界铁律**:agent 只做推理/判断,代码只做机械/确定/状态。
2. **盯支点,不盯盘面**:卡的核心是**支点(判断所依赖的命门事实)**,不是价格,也不是软绵绵的"假设"。
3. **事实与判断分离**:卡内分区存放;事实可复算可刷新,判断必须挂在事实(支点)上、可追溯。
4. **专业分工靠委托**:一张卡,不同内容由主 agent 委托宏观/行业/标的**专业 subagent**采集。
5. **硬规则走代码**:纪律信号(排雷/分位/逆向DCF/硬证伪)= **钉死、可测、版本化的 `invest_lib`**,由 **builtin compute 工具**执行(§8.2);**绝不让 LLM 每次现写阈值/公式**。其余(行业-specific、探索性计算)才放开给 agent 脚本。

---

## §1 核心理念:单标的卡闭环

整个系统只有**一个一等实体:标的卡**。一只股票一张卡 = **一条 append-only 时间线**,由不可变诊断快照(**事实区 + 判断区**两区,判断内含支点)+ 决策/复盘条目组成;**当前态 = 最新快照**。闭环:

```
①建立 ──→ ②逻辑跟踪 ──→ ③决策 ──→ ④复盘 ──┐
  ↑                                          │
  └────────── 复盘反哺下一次建卡 ←───────────┘
```

宏观/行业**不再是独立的卡**,而是**标的卡事实区里的两个事实块**(由专业 subagent 采集回填)。系统从"多实体 + 传导引擎"压成"单实体 + 一条时间线"。

## §2 数据模型:快照时间线 + 两区

### 2.1 卡 = 一条不可变快照的时间线

逻辑卡的本质是**某一时刻的诊断快照,immutable**。一张标的卡 = **卡头 + 一条 append-only 时间线**,时间线上只有三类条目:

```
StockCard
├─ 卡头   id / code / name / status(watching|held|closed) / created_at / updated_at
│         id = 卡标识「代码-日期-唯一」(如 002594-20260612-3f9a2b1c);**一股可多卡**;
│         所有读写严格按此精确 id(create_card 返回,原样透传)。code = 股票代码(可按它列同股多卡)。
└─ 时间线(append-only,三类条目)
     snapshot   诊断快照(建立/每次复查)  ← 两区 + delta_vs_prev,immutable
     decision   决策(基于某快照的操作)    ← ref 它依据的 snapshot
     review     复盘(跨快照+决策的反思)    ← ref 它复盘的 snapshots/decisions
```

复查不修改旧快照,而是**产出一个新快照**;决策/复盘**引用快照、不复制内容**。天然满足 immutable / append-only / point-in-time 可重放 / 无后视镜偏差(look-ahead bias)。

### 2.2 诊断快照 snapshot(两区:事实 + 判断,支点在判断里)

```
Snapshot {
  at
  ├─ 【事实区 facts】客观·可溯源,每条是 Fact;分三块,由对应 specialist 采
  │   ├─ stock     标的事实(业务/财务/估值/治理)
  │   ├─ industry  行业事实(景气/格局/估值/政策)
  │   └─ macro     宏观事实(货币/流动性/增长通胀/制度)
  └─ 【判断区 judgments】agent 推理结论;每条判断声明它压在哪些事实上(= 支点)
  + delta_vs_prev?  相对上一快照的变化(= 跟踪结论)
       { fact_changes[], fulcrum_shifts[], rejudges[] }
}
```

**Fact(事实 = 当前值 + 多窗口趋势 + 来源)**:

```
Fact {
  kind:    'hard' | 'soft'             // 硬数值 / 软定性
  current: { value, as_of }            // 数值或定性结论 + 数据时点
  source:  Source                      // 具体数据源地址(回看溯源 + 复查前探,见 §2.4)
  trend?:  [ { window, change, change_pct, direction } ]   // 仅硬数值,多窗口
  series_ref?: string                  // 原始序列在卡工作区的引用(不内联进 LLM)
}
```

**Judgment(判断 = 结论 + 支点)**——支点不是独立的区,而是**判断声明的命门事实**:

```
Judgment {
  key:        business_logic | financial_logic | management_logic | valuation_view | thesis | action_advice
  conclusion: 结论(agent 推理产物)
  fulcrums:   [                        // 支点 = 这条判断压在的命门事实(必 ≥1)
    { id,                              // 支点 id(供 decision.ref_fulcrums / 跟踪去重引用)
      ref,                             // ★ 指明对应的那条事实(指向事实区,必填)
      statement,                       // 如 "毛利率≥30%"
      kind:    'hard' | 'soft',
      trigger,                         // hard:{op,value,persistence}(代码比对);soft:自然语言变化条件
      status:  'holding' | 'shaken' | 'broken',
      last_checked,
      corroboration? }                 // 软支点的多源印证数(承重需 ≥2,见 §8.6)
  ]
}
```

要点:

- **支点 = 判断依赖的命门事实**,每个**必 `ref` 指向事实区一条事实**(指明对应事实)——支点不漂浮、不自造。
- **判断必含支点**(`fulcrums` 非空),否则是"空中楼阁",拒绝落库(§11)。
- 一条事实可被多条判断引为支点(各自 ref);跟踪按事实去重盯。
- 支点 `trigger`/`status`:hard 走代码比对、soft 走 agent 复读复核(§4.2)。
- **窗口自适应**:日频事实(价格/PE·PB分位/换手/北向)用 5/10/30/60 交易日;报告期(财务)用近 N 报告期(QoQ/YoY)。趋势由 `invest_lib.computeTrend` 算(§8)。

### 2.3 决策 / 复盘条目(引用快照,不复制内容)

```
decision { at, action:建|加|减|清, price, weight, reason*, ref_snapshot, ref_fulcrums[] }
review   { at, scope, thesis_realized, attribution:{对错 × 运气|能力}, lesson, ref_snapshots[], ref_decisions[] }
```

- **判断 vs 决策**:判断(snapshot 内,agent 的看法+建议)是"**建议**";决策(timeline 条目,用户行动)是"**行动**"。`action_advice`(建议方向)≠ `decision.action`(实际执行)。决策**基于判断但用户自主**(可 override 建议,agent 只在"决策刹车"质询一致性、不替用户下单——非荐股 §11)。
- **`ref_snapshot` = 行动锚定的逻辑片段**:决策钉在它做出时那一刻的诊断快照上 —— **复盘衔接的锚点**(没有它,复盘不知道行动当初基于什么逻辑)。
- **决策 `reason` 必填**(`*`):无理由的 decision **拒绝落库**(代码强制,见 §11)。
- **卡 status 转换**:建卡 → `watching`;建/加仓 decision → `held`;清仓 decision / 触发一票否决 → `closed`。
- 时间线可按类型切片查询(所有 decision / 所有 review),不丢检索性。
- **存储**:MVP 每次复查存一份完整快照(几 KB)即可;去重优化留 v2。

### 2.4 数据源地址与诊断衔接

每条 Fact 的 `source` 是**具体、可复用的地址**,同时服务两个用途:**回看(溯源到原文)+ 前探(下次诊断增量重取)**。

```
Source {
  channel: 'mcp' | 'web'
  metric_key?: string          // 硬: 稳定语义键(如 'gross_margin')——复查靠它解析到当前 MCP,不写死地址
  // —— 硬数值(mcp)——
  endpoint?: '<server>/<tool>' // 本次实际地址,如 'akshare-one-mcp/get_financial_indicator'(溯源/重放)
  params?:   { symbol, period }// 调用入参 → 复查重调拿最新
  // —— 软定性(web)——
  url?:      '<原文 URL>'      // 回看定位
  doc_id?:   '<公告号/研报编号>'// 规范定位(URL 失效仍可找回)
  publisher?: string  tier?: T1|T2|T3   // 机构/媒体 + 来源分级(§8.6)
  query?:    '<检索式模板>'    // 复查按此检索 as_of 之后的新进展
  // —— 共通 ——
  as_of:     '<数据时点>'      // 财报期/公告日 = 增量水位
}
```

| 用途               | 硬数值                                                        | 软定性                                                 |
| ------------------ | ------------------------------------------------------------- | ------------------------------------------------------ |
| **回看**(溯源)     | `endpoint + params` 复算/重放                                 | `url` / `doc_id` 回到原文                              |
| **前探**(复查重取) | `metric_key` 解析到**当前配置的数值 MCP**(可替换)→ 重调拿最新 | 按 `query` 检索 `as_of` **之后**的新进展(不重读旧 url) |

**衔接 = 双水位增量**:复查读 `source.as_of`(数据多新)+ 支点 `last_checked`(我多久没看)→ 按地址从水位往前增量重取 → 三种结果:

- **无新数据**(硬:as_of 没前进 / 软:无 as_of 之后的新文档)→ 事实原样进新快照,只更支点 `last_checked`
- **数据变了** → 新 source(新 as_of / 新 url)+ 记 `fact_change`,可能触发 `fulcrum_shift`
- **复核仍成立**(查了、确认没变)→ as_of 不变、`last_checked`=now(区别于"压根没查")

> 换源提醒:若两次快照间数值 MCP 被替换,新 `endpoint` 变了 → provenance 变化,`cross_check` 老/新源避免"换源造成的假变化"。详细复查流程见 §7②。

## §4 支点机制(判断的命门,卡的灵魂)

### 4.1 支点 = 判断依赖的命门事实(指向事实,不是假设)

> **假设式(弱)**:"我假设需求会持续高增长" — 预测,无法干净证伪。
> **支点式(强)**:支点 = 判断**真正压在的那几条事实**——它们变了,判断就得推翻重来。

事实有几十条,但一条判断只**重度依赖少数几条命门事实**;那几条 = 这条判断的支点(每个 `ref` 指向事实区一条事实)。标支点只为一件事:**跟踪只盯命门,不用盯全部**(盯支点不盯盘面)。

> 中际旭创·"值得持有"这条判断,命门支点 =「①北美 capex 指引 ②800G 出货占比 ③毛利率≥30% ④CPO 未量产」——全是当前事实;负债率、某日北向流入等几十条不那么要命的,不进支点。

主 agent 写判断时**为每个支点定"变化阈"**(多少变化算逻辑塌)——这是支点里唯一的判断成分,事实本身仍客观。

### 4.2 证伪 = 支点对应事实变化 → agent 推理裁定(代码只给确定性信号)

> **证伪是 agent 的推理结论,不是代码直接判定。** 代码给"阈破没破"这个确定性事实(§0.5),agent 给"是否真的证伪了逻辑"这个判断(§0.1)。流程:**推理(查哪个支点)→ 获取事实 → 计算(确定性信号)→ 推理裁定(支点状态)**。

| 环节                | hard 支点(数值)                                  | soft 支点(定性)             |
| ------------------- | ------------------------------------------------ | --------------------------- |
| ① 推理要查什么      | agent 按支点 trigger                             | agent 按支点描述            |
| ② 获取事实          | 数值 MCP 重取(按支点 ref / metric_key)           | web 检索 as_of 之后         |
| ③ 计算/检出(确定性) | `check_falsification`/`compute_trend` → 阈破没破 | (无计算,返回文本)           |
| ④ **推理裁定**      | **agent** 据信号 + 上下文 → 支点状态             | **agent** 读文本 → 支点状态 |

**状态边界(三态各司其职)**:阈破 / 检索到异动 → 支点自动标 **`shaken`(机器警报,待确认)**;**agent 推理裁定** → `broken`(证伪成立、逻辑塌)或回 `holding`(确认是一次性/噪音)→ 并重判该支点所撑的判断。

> 例:毛利率单季跌破 30% → `check_falsification` 触发 → `shaken`;agent 查明是一次性产线搬迁成本 → 裁定回 `holding`(不算证伪)。代码不擅自宣判,agent 不绕过计算。

### 4.3 卡内轻量"传导"(取代跨卡传导)

复查产出新快照时一次性完成,记入该快照的 `delta_vs_prev`:

```
重取事实 → fact_changes → 阈破/异动 → 支点 holding→shaken(代码,机器警报)
   → agent 裁定 broken/回 holding(fulcrum_shifts)
   → 反查"哪些判断把它列为支点" → 主 agent 重判那些判断(rejudges,可改可维持)
```

因为支点就活在判断里(`judgment.fulcrums`),"支点变 → 反查所撑判断"是直接遍历,无需独立的支点层。

### 4.4 典型支点(辅助选取)

成长股:营收增速/毛利率/大客户订单/行业景气价格/产能利用率/行业渗透率(空间未到顶) · 价值股:股息率/OCF/PB分位/资产质量 · 周期股:行业价格/库存周期/产能投放。

## §5 判断区(agent 推理的结论,每条挂支点)

> **判断 = `推理→获取→计算→推理结论` 循环的产物,与证伪裁定(§4.2)同构** —— 不是对事实区的被动读取。agent 推理"这条判断要看什么" → 获取(MCP/web)→ 计算(compute 工具)→ 推出结论,并**声明它压在哪些命门事实上(= 支点,每个 ref→事实)**。**代码 owns 获取/计算(事实+信号),agent owns 推理两端(提问 + 下结论 + 选命门)**。

| 判断             | 推理结论(内容)                                | 命门支点(ref→事实)                           |
| ---------------- | --------------------------------------------- | -------------------------------------------- |
| business_logic   | 业务逻辑:增长来源、护城河、商业模式、成长空间 | 营收增速 / 大客户 / 行业空间 / 产业链卡位    |
| financial_logic  | 财务逻辑:盈利质量、现金流、排雷结论           | OCF含金量 / 8红灯 / 关键财务比率             |
| management_logic | 管理逻辑:**一票否决项**(治理出红线直接弃)     | 质押 / 违规 / 资金占用 / 控制人              |
| valuation_view   | 估值判断:贵/便宜、买卖点区间                  | PE分位 / 逆向DCF隐含增速 / 可比横比 / 预期差 |
| thesis           | 综合结论:为什么值得持有/观察                  | 上述判断的核心支点汇总                       |
| action_advice    | 操作建议:建/加/减/清 + 理由                   | 核心支点 + valuation_view                    |

- **每条判断 `fulcrums` 必非空**(无支点拒绝落库)——支点即"这条判断的命门事实引用"。
- **(派生)支点视图**:把全部判断的支点**按事实去重汇总** = 这张卡的命门清单 + 逻辑健康度(跟踪/UI 用,§9.4),非单独创作的区。

## §6 Agent 架构与建卡推理

> **agent 的核心操作循环 = `推理 → 获取 → 计算 → 推理结论`**:推理"要什么" → 获取(数值 MCP / web 检索)→ 计算(compute 工具,确定性)→ 推出结论。**代码 owns 中间两步(事实 + 信号),agent owns 两端(提问 + 下判断 + 选命门支点)**。事实区、支点裁定(§4.2)、判断(§5)都是这同一循环的产物;§6.3 建卡推理模板是它在"建卡"上的具体展开。

### 6.1 主-专家委托:specialist 是"带工具箱的研究员"

```
                 ┌─────────────────────────────┐
   用户 ◄──────► │  主 agent  标的研究(判断者)   │
                 │  · 委托三专家采事实           │
                 │  · 写判断 · 为判断选命门支点  │
                 │  · 复查(跟踪模式)            │
                 └───┬─────────┬─────────┬───────┘
       delegate_agent│         │         │
            ┌────────▼──┐ ┌────▼─────┐ ┌─▼────────┐
            │宏观 specialist│ │行业 specialist│ │标的 specialist│   ← 研究员:判断要什么数据→取数→算/读→交事实
            └───────────┘ └──────────┘ └──────────┘

            ┌──────────────────────────┐
            │  复盘 agent(审计者)        │   ← 独立,不为原逻辑辩护
            └──────────────────────────┘
```

**specialist 的工作方式 = 研究员,不是流水线消费者**:

1. **判断要什么数据**(走推理模板 §6.3:拆盈利 + pre-mortem + LLM 自身行业知识 —— 银行看净息差/不良率,制造看产能/订单)。
2. **路由取数手段**:硬数值 → 调**数值 MCP**(LLM 直调,可替换);软定性 → **web 检索**(行业网站/公告/研报)。
3. **算 / 读**:硬数值 → 喂 **6 个 builtin compute 工具**(铁律,确定性)或 `bash` 脚本(行业-specific);软定性 → 读文本提炼,必要时再调 compute 得结论。
4. **交回结构化事实块**(每条 Fact 带 kind/source)。

> **specialist 只采事实、主 agent 才下判断并选命门支点** —— 同时满足"专业委托"(§0.4)与"事实/判断分离"(§0.3)。复盘独立,守证伪/复盘的诚实纪律。Talor 落地走 `delegate_agent`(隔离、一级、预算约 3/session,正好三专家)。

### 6.2 每个 agent 的配合规格

| agent               | 角色        | 用的系统能力                                   | 产出                            | 代码守的边界                    |
| ------------------- | ----------- | ---------------------------------------------- | ------------------------------- | ------------------------------- |
| **主 agent**        | 编排 + 判断 | `delegate_agent` + card 工具(读写)             | snapshot(写判断 + 选命门支点)   | 判断必含支点(ref→事实);不自取数 |
| **标的 specialist** | 采标的事实  | 数值MCP + web检索 + **6 个 compute 工具**      | 标的事实块(业务/财务/估值/治理) | 只采不判;铁律必调 compute       |
| **行业 specialist** | 采行业事实  | 数值MCP + web检索 + `compute_*`                | 行业事实块                      | 同上;受限视图                   |
| **宏观 specialist** | 采宏观事实  | 数值MCP + web检索 + `compute_trend/percentile` | 宏观事实块                      | 同上;受限视图                   |
| **复盘 agent**      | 审计归因    | card 工具(读时间线)                            | review 条目                     | 独立,不为原逻辑辩护             |

### 6.3 建卡推理模板(核心:agent 如何自己推出"关键信息")

**不给行业清单(content,要厚 skill),给推导方法(method,几乎零 skill)。** agent 对任何票,用通用推理动作推出关键信息:

| 动作                 | 问什么(含 4 个深度追问)                                                           | 产出                                                   |
| -------------------- | --------------------------------------------------------------------------------- | ------------------------------------------------------ |
| ① **拆盈利公式**     | 靠什么赚钱? + **增量从哪来(成长来源)?空间还多大(TAM/渗透率)?产业链谁议价(卡位)?** | 驱动因子 + 成长来源 + 行业空间 + 产业链卡位            |
| ② **Pre-mortem**     | 怎么崩? + **近期有什么正向验证点(催化剂)?**                                       | 命门支点候选 + 证伪条件 + 催化剂日历                   |
| ③ **三维 checklist** | 业务/财务/管理 都覆盖了吗?                                                        | 补漏(管理=一票否决)                                    |
| ④ **估值反推**       | 隐含什么预期? + **vs 可比公司贵不贵?**                                            | `reverse_dcf` 隐含增速 vs 一致预期 → 预期差 + 可比横比 |

**同一方法推出不同行业的关键**(无需 skill 列举):

| 行业   | ① 拆盈利 → 关键事实                                 | ② pre-mortem → 命门支点        |
| ------ | --------------------------------------------------- | ------------------------------ |
| 银行   | 规模×净息差−信用减值 → 净息差/不良率/规模           | 息差收窄、不良飙升、监管       |
| 光模块 | 出货量×(单价−成本) → 800G出货/产能/毛利/大客户capex | CPO替代、价格战、capex下修     |
| 消费   | 门店×单店×净利率 → 渠道/同店/复购/毛利              | 渠道见顶、品牌力衰退、成本上行 |

**建卡推理序列(嵌在建卡引导/极薄 prompt,非 skill)**:

```
对任何票走一遍:
 1. 拆盈利公式 → 列驱动因子 + 成长来源(增量从哪)+ 行业空间(TAM/渗透率)+ 产业链卡位
 2. 每个因子:能拿数值吗? → 标 数值MCP / 定性web
 3. Pre-mortem → 列风险 → 命门候选 + 证伪条件;顺带列近期催化剂(正向验证点 + 时点)
 4. 三维(业务/财务/管理)过一遍 → 补漏
 5. 估值:隐含什么预期? reverse_dcf → 预期差;选 2~3 可比公司横比(PE/增速/ROE)
 6. 取数 → compute算/读文本 → 填事实区 → 写判断,并为每条判断挑命门事实标为支点(ref→事实 + 阈) → snapshot
```

> 这是**推理模板(method)**,通用、极小、跨行业;"卡片驱动 agent"在此落地——agent 照序列推,自己推出每只票的关键信息与命门支点。

### 6.4 skill 淡化:卡即指令(self-describing)

方法论从"散文 skill"迁到四处,skill 淡化到只剩冷启动 hint:

| 原本要写进 skill 的                 | 迁到                                                                                               |
| ----------------------------------- | -------------------------------------------------------------------------------------------------- |
| 两区结构、建卡流程                  | **卡 schema / 模板本身**(结构即流程)                                                               |
| 某槽位用哪个工具取/算               | **slot 元数据**:`{key, zone, acquire:'numeric_mcp'\|'web', compute?, required, fulcrum_candidate}` |
| 行业要看什么                        | **拆盈利+pre-mortem 推理模板(§6.3)+ LLM 自身知识**                                                 |
| compute 工具怎么用                  | **每个工具的 `description` 字段**(Talor 原生)                                                      |
| 铁律(必调compute/判断挂支点/reason) | **代码守护点(§11)**                                                                                |

- **跟踪自驱**:已建卡的判断已写明"支点 ref→事实 + trigger" → 复查时**卡自己就说清要重取什么、怎么比对**(`check_falsification` 读支点的 trigger),零 skill。
- **代码完整性兜底**:`append_snapshot` 校验 **三维齐全 + 每条判断含支点 + 每个支点 ref→事实且有 trigger + required 槽位非空** → agent 推漏被退回自我纠正。
- **残留**:极冷门/特殊商业模式,模板留**一句话级行业 hint** 做冷启动锚点(非完整 skill)。

## §7 闭环四动作(完整流程)

### ① 建立(首诊)— 主 agent

1. 用户给定标的 → 主 agent 并行 `delegate_agent` 三专家。
2. 各 specialist 走**建卡推理模板(§6.3)** → 调数值 MCP / web 检索 → 调 compute 工具 / 读文本 → 交事实块。
3. 主 agent 汇总事实区 → 写三维逻辑 + 估值判断 + thesis,**为每条判断挑命门事实标为支点(ref→事实 + 变化阈)**。
4. 落首个 **snapshot**(status=watching 或 held)。

### ② 逻辑跟踪(复查)— 主 agent(跟踪模式),周期 / 事件触发(MVP:打开时主动复查;后台 push → v2)

> 跟踪 = "命门事实变了"。不是单独记录,而是**产出一个新 snapshot**,变化记入 `delta_vs_prev`。

1. 遍历各判断的支点,按其 `source` 地址 + 双水位(`as_of` / 支点 `last_checked`,见 §2.4)**增量重取对应事实**:hard 用 metric_key 解析当前 MCP 重调、soft 按 query 检索 as_of 之后。
2. hard 支点 → `check_falsification`/`compute_trend` 给阈破信号(标 `shaken`);soft 支点 → agent 检索复读。
3. **agent 推理裁定支点** `broken`/回 `holding`(§4.2);broken/shaken → 反查所撑判断 → 重判(可改可维持)。
4. 写新 **snapshot**(含 delta);若证伪触发 → 提示决策。

### ③ 决策 — 用户决策,agent 给建议

1. 主 agent 基于最新 snapshot → 给 action_advice(建/加/减/清 + 理由 + 当时估值/价格)。
2. **决策刹车(§9.4)**:落库前 agent 对账「当初支点 vs 现在」+ 逼问理由 + **情绪型理由软拦**(可 override,照实记录);质询一致性不评方向(非荐股)。
3. 用户决策 → 落 **decision** 条目(ref snapshot,记 price/weight)。**`reason` 必填**。

### ④ 复盘 — 复盘 agent(独立),平仓后 or 周期

1. **逐个行动反推**:取每个 decision 的 `ref_snapshot`(行动时的逻辑)→ 对照其后快照的支点/事实演变 + 实际结果。
2. **对过程不对结果**:判"行动有没有踩对当时的逻辑",而非只看盈亏 —— 赚了可能是错过程靠运气,亏了可能是对过程没兑现。
3. **四象限归因 = 「行动顺/逆逻辑」×「结果好/差」→ 能力 / 运气**;结合决策当时的 `reason`(含情绪型标记)定责。
4. 落 **review** 条目 → 沉淀 lesson → 反哺下次建卡的支点选取与证伪设置。

### §7.5 关键流程时序与环节说明

**A. 建立(首诊)端到端时序**

```
用户: "研究 300308 中际旭创"
  │
主agent ──delegate──┬─► 标的 specialist ─┐
                    ├─► 行业 specialist ─┤  (并行,各自带受限工具/MCP视图)
                    └─► 宏观 specialist ─┘
                              │ 每个 specialist 内部:
                              │  1) 判断要什么(走推理模板 §6.3:拆盈利+pre-mortem)
                              │  2) 调数值 MCP(可替换)→ 读数值结果   (硬数值)
                              │     web_search/fetch_url → 文本        (软定性)
                              │  3) 字段喂 compute_* 工具 → 红灯/分位/DCF/trend (铁律,确定性)
                              │     读文本 → 提炼软事实(必要时再调 compute 得结论)
                              │  4) 交回事实块(每条 Fact{kind,current,trend,source})
                              ▼
主agent: 汇总事实区 → 写三维逻辑/估值/thesis,每条判断挑命门事实标支点(ref→事实 + 阈)
  │
  └─► append_snapshot → 时间线首个 snapshot(status=watching|held)
```

**B. 跟踪(复查)端到端时序**

```
触发(MVP:打开时主动复查)
  │
主agent(跟踪模式): 遍历各判断的支点(去重 by 事实)
  │  对每个支点(按 §2.4 source 地址 + 双水位增量重取):
  │   hard → metric_key 解析当前 MCP 重取序列 → check_falsification 比对 → 阈破=标 shaken(代码)
  │   soft → 按 query 检索 as_of 之后 → agent 复读 → 异动=标 shaken
  ▼
agent 推理裁定:shaken → broken(证伪成立)或回 holding(一次性/噪音)
  │ → broken/shaken 时反查"哪些判断把它列为支点" → 主agent 重判(可改可维持)
  ▼
append_snapshot(新快照 + delta_vs_prev{fact_changes,fulcrum_shifts,rejudges})
  │ 若 broken/重大变化 → 输出 action_advice 提示用户决策
```

**C. 关键环节说明(每个受守护的节点 = 一条纪律)**

| 关键环节       | 做什么                                                | 谁守                  | 守什么(对应铁律)                             |
| -------------- | ----------------------------------------------------- | --------------------- | -------------------------------------------- |
| **数值取数**   | 调数值 MCP 取数,LLM 读结果                            | MCP(LLM 直调)         | 数据源可替换(§8.1)                           |
| **铁律计算**   | 红灯/分位/DCF/证伪只走 compute\_\* 工具               | 代码(invest_lib)      | 硬规则走代码、可复现(§0.5)                   |
| **自定义计算** | 行业指标用 bash 脚本,脚本存进快照                     | agent + bash          | 灵活性 + provenance 可复现                   |
| **选命门支点** | 为每条判断挑承重事实 + 定阈,ref→事实                  | 主agent 判断          | 盯支点不盯盘面(§0.2)                         |
| **判断挂支点** | 每条判断 fulcrums 非空且 ref→事实                     | 代码校验              | 无支点拒绝落库(§11)                          |
| **证伪裁定**   | 代码给阈破信号(shaken)、**agent 裁定 broken/holding** | compute 工具 + agent  | 任务边界:代码给事实、agent 给判断(§0.1/§4.2) |
| **卡内传导**   | 支点变 → 反查所撑判断 → 重判,记 delta                 | 代码反查 + agent 重判 | 变了才动判断、可追溯(§4.3)                   |
| **决策落库**   | reason 必填,ref snapshot                              | 代码校验              | 决策必注明理由(§11)                          |
| **快照不可变** | 复查产出新快照,不改旧                                 | 代码(append-only)     | point-in-time、无后视镜(§2.1)                |
| **复盘独立**   | 复盘 agent 与建卡 agent 分离                          | agent 架构            | 不为原逻辑辩护(§6)                           |

## §8 取数 / 计算 / 判断 三分(核心机制)

三层职责清晰分离:**数据 = MCP(可替换)/ 计算 = builtin compute 工具(方法论 IP)/ 编排 = LLM**。流程:**LLM 调 MCP 取数 → 读结果 → 调 compute 工具(传入字段)算硬信号 → 组装进卡**;软信息走 web 检索由 agent 读。

> LLM 充当 adapter:换数据源时 LLM 读不同 shape、仍喂同一个 compute 工具 → **零适配代码,任何 MCP 热插拔**。计算逻辑(阈值/公式)始终在 compute 工具(代码)里 —— LLM 中转的是**数据值**,不是**规则**。

### 8.1 数值数据:MCP 供给(语义契约,源无关、完全可替换)

代码**不绑任何 MCP 工具名**;agent 由推理模板(§6.3)推出"需要哪些数据项",调当前配置的数值 MCP 取、读结果、喂 compute 工具。换源只要新源能供这些**语义项**,compute 工具与卡片结构不变。

| 卡片块 | 数值项                                                                          | 形态      | 去向                                  |
| ------ | ------------------------------------------------------------------------------- | --------- | ------------------------------------- |
| 业务   | 营收/归母净利/扣非净利、毛利率/净利率                                           | 近8期序列 | 业务事实 + `compute_trend`(YoY)       |
| 财务   | OCF、应收(+YoY)、存货(+YoY)、商誉、净资产、有息负债、总资产、货币资金、审计意见 | 当前/序列 | `compute_red_flags` 入参              |
| 财务   | ROE/ROIC、资产负债率、利息保障倍数                                              | 近5年     | 财务事实                              |
| 估值   | PE(TTM)/PB 历史序列(近5年日频)、股息率历史                                      | 序列      | `compute_percentile`                  |
| 估值   | 当前 PE/PB/PS、EPS、当前价、总/流通市值                                         | 当前      | `reverse_dcf` 入参 + 估值事实         |
| 行业   | 行业 PE/PB 历史序列、行业营收利润增速                                           | 序列      | `compute_percentile` + 行业事实       |
| 宏观   | 利率(MLF/LPR/逆回购/10Y)、M2/社融、CPI/PPI、PMI、GDP、北向净流入、汇率、全A PE  | 当前/序列 | 宏观事实 + `compute_trend/percentile` |
| 治理   | 质押比例、解禁时间表                                                            | 当前      | 治理事实                              |

### 8.2 内置计算工具(6 个,= invest_lib 逐个暴露)

`compute/` 纯函数逐个包成 builtin tool。每个 take 结构化入参(LLM 从 MCP 结果里填)→ 返回信号:

| 工具                  | 入参                                                                                                                                       | 出参                                                | 产出卡片字段                           |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------- | -------------------------------------- |
| `compute_red_flags`   | `{ocf, net_income, deducted_ni_history, rev_yoy, ar_yoy, inv_yoy, goodwill, equity, interest_debt, total_assets, cash, audit_nonstandard}` | `RedFlag[]`(8 条)                                   | 财务事实.red_flags(排雷 + 一票否决)    |
| `compute_percentile`  | `{series:number[], current}`                                                                                                               | `0~1 \| null`                                       | PE/PB/股息/行业 历史分位               |
| `reverse_dcf`         | `{price, eps, exitPe, horizonYears, discount}`                                                                                             | `{impliedGrowth, reasonable}`                       | 估值事实.逆向DCF隐含增速               |
| `compute_trend`       | `{series:number[], windows:number[]}`                                                                                                      | `[{window,change,change_pct,direction}] + momentum` | 任何 Fact.trend(5/10/30/60 或 QoQ/YoY) |
| `check_falsification` | `{rule:{metric,op,value,persistence}, series:number[]}`                                                                                    | `{triggered, checkedPeriods}`                       | 跟踪:hard 支点是否破阈                 |
| `cross_check`         | `{values:[{value,source}], tolerance}`                                                                                                     | `{value, consistent, note}`                         | 多源可靠性(存疑标记)                   |

- 简单派生(PEG=PE÷增速、应收−营收增速 gap)无需单独工具:gap 已在 `red_flags` 内,PEG 类一步除法 LLM 直接算。**这 6 个就是方法论 IP 的全部确定性计算。**
- **长序列 fallback**:`compute_percentile`/`compute_trend` 需 60+ 点,若 LLM 转写有负担 → 支持 `series_ref`(MCP 结果存盘、传引用,序列不经 LLM 转写)。先不做,留作优化。

### 8.3 信息类(软)数据:web 检索 → 推理/(调计算)→ 结论 → 落卡片

**纪律:每次检索都对准一个卡片字段/支点,不漫无目的搜。** "推理 + 必要时调 compute" 产出可落卡的结论:

| 卡片软信息                          | 检索目标               | agent 处理                                                  | 落卡片字段                                    |
| ----------------------------------- | ---------------------- | ----------------------------------------------------------- | --------------------------------------------- |
| 行业地位/份额/竞争格局              | 研报/年报/行业网站     | 读 + 推理                                                   | `business_logic` + 业务软事实                 |
| 护城河/商业模式                     | 年报/研报              | 推理                                                        | `business_logic`                              |
| 大客户关系/订单动向                 | 公告/新闻/研报         | 读                                                          | business_logic 的 soft 支点                   |
| 高管变动/大股东增减持/违规/关联交易 | 公告/交易所            | 读                                                          | 治理软事实 → `management_logic`(一票否决候选) |
| 行业景气方向/拐点                   | 行业网站/价格数据/研报 | 读 +(若得价格序列)调 `compute_trend`                        | 行业软事实 + 景气趋势                         |
| 产业政策/监管动向                   | 政府网/政策检索        | 读 + 推理(顺风/逆风/一票否决)                               | 行业/宏观 政策事实                            |
| 催化剂                              | 公告/研报              | 读                                                          | `thesis` 支点                                 |
| **一致预期(预期差)**                | 研报                   | 读市场一致增速 → **调 `reverse_dcf` 算隐含增速 → 两者对比** | `valuation_view`.预期差                       |

> 最后一行 = "推理 + 调计算给结论"的典型:软信息(一致预期)+ 硬计算(reverse_dcf)→ agent 得出"是否 price-in / 有无预期差"。

### 8.4 数据流(LLM 编排)

```
specialist(标的, 以 600036 为例):
  ① 调数值 MCP: get_financial_indicator('600036') → 读结果
  ② compute_red_flags({ocf, net_income, ...从结果填})        → 8 红灯
     compute_percentile({series:PE历史, current:当前PE})       → PE 分位
     reverse_dcf({price, eps, exitPe, ...})                    → 隐含增速
  ③ web_search('招商银行 资产质量 监管 净息差') → 读文本       → 软事实
     [行业项] bash 脚本算自定义指标(读 MCP 落地数据)           → 自定义数值
  ④ 组装事实块(每条 Fact{kind, current, trend, source})→ 交主 agent
主 agent: 写判断 + 选命门支点(ref→事实) → append_snapshot
```

### 8.5 覆盖校验(卡片每块事实都有来源,无悬空)

| 卡片事实块                | 来源路径                              |
| ------------------------- | ------------------------------------- |
| 业务/财务/估值/治理(数值) | 数值MCP → compute 工具                |
| 业务/治理(定性)           | web检索 → agent 读                    |
| 估值(预期差)              | 研报检索 + `reverse_dcf` → agent 对比 |
| 行业(估值/增速)           | 数值MCP → `compute_percentile`        |
| 行业(景气/政策/格局)      | web检索 →(+compute_trend)→ agent      |
| 宏观(利率/M2/北向…)       | 数值MCP →(+compute_trend/percentile)  |
| 宏观(制度/监管)           | web检索 → agent                       |

✅ §3 每块都有明确来源:**硬 = 数值MCP + compute 工具;软 = web检索 + agent(必要时调 compute)**。

### 8.6 软信息可靠性(来源分级,对称于硬数据的 cross_check)

硬数据靠 provenance + 多源 `cross_check`;软数据(检索)靠**来源分级 + 一手优先 + 承重多源**。

**只允许 T1/T2/T3,T4 及以下一律不采(连线索都不取)**:

| Tier                | 来源                                                | 用途                |
| ------------------- | --------------------------------------------------- | ------------------- |
| **T1 一手官方**     | 公告/定期报告、交易所、证监会、央行、统计局、政府网 | 事实首选            |
| **T2 权威二手**     | 券商研报(具名)、行业协会、权威数据商                | 事实可用(注明机构)  |
| **T3 主流财经媒体** | 财新/第一财经/证券时报等                            | 可用,优先找一手印证 |
| **禁(T4↓)**         | 一般媒体/自媒体/股吧/论坛/匿名/营销号               | 不采                |

规则:① **一手优先**(T1>T2>T3,有官方原文必取一手);② **承重软支点 ≥2 独立源印证**(`corroboration≥2`),否则标「单源存疑」;③ **事实/观点分离**(研报预测/评级是观点,标"X机构观点",不当事实);④ **带发布日期**,过时标 `stale`;⑤ **可溯源防幻觉**,结论必 cite 原文,未找到如实说、不编。

强制:**检索工具(web_search/检索 MCP)配域名分级表,只返回 T1-T3**;`append_snapshot` 校验软支点 `source.tier ∈ {T1,T2,T3}`、`corroboration≥2`,否则拒绝/告警。

> v2 逃生口:个别高质量自媒体走「**具名加白**」单独放行,而非开放整个 T4。

### 9.4 UI 与交互(常驻案卷 + 三主动介入)

**定位**:不是"数据看板",而是**纪律 + 记忆工具**——补散户"管不住情绪、记不住教训"两短板。由此推出两个硬要求:**持久可回看的案卷(记忆)** + **关键时刻主动介入(纪律)**。

**布局:左对话(引擎)+ 右常驻案卷(记忆)**

```
┌──── 对话(研究/问答/复盘,Talor 原生)────┬──── 常驻案卷(右侧)────┐
│ agent 流式研究 → 建好 [▸卡 chip]          │ [主显示区] 选中节点的完整卡 │
│ ⚠[逻辑警报 chip]                          │  as_of · 事实 · 判断(挂支点) │
│ [决策对账块:当初支点 vs 现在 + 理由]       │ ───────────────────────── │
│                                           │ [时间线轨] ●建 ●查 ◆决 ★盘 │
└──────────────────────────────────────┴───────────────────────┘
```

**一张卡三密度,上下文决定**(渐进式披露):

- **Chip(收起)**:对话流默认——标的 + status + **逻辑健康度 🟢🟡🔴**(= 全部支点状态汇总)+ 一句 thesis;旧卡自动折叠,不刷屏。
- **Card(展开)**:点 chip → 完整两区(事实折叠 / 判断展开,每条判断下列其命门支点 + 状态色)。
- **案卷(常驻)**:右侧持久——**单一真相**,两区 + 时间线;对话里只需 chip 引用。

**案卷 = 时间线轨 + 主区(时光机)**:

- 默认主区 = 最新快照(=当前态);**点历史节点 → 主区切到该快照(只读 + 标 as_of)**,"回到当前"返回。全快照存储(§2.3)→ 跳转即渲染。
- 三类节点:`● snapshot`(建卡/复查,摘要=逻辑健康度变化)/ `◆ decision`(加减@价 + 理由;**情绪型理由标红**)/ `★ review`(四象限 + lesson)。
- **查看历史快照只见当时已知信息** → §11"无后视镜"在 UI 上自动成立。
- **不做快照可视化对比**(留 v2);决策对账由 agent 文字呈现"当初 vs 现在",复盘=顺节点逐张跳看。

**派生支点视图**:案卷把**全部判断的支点按事实去重汇总**成一个"命门清单 + 健康度"(🟢holding/🟡shaken/🔴broken),供一眼扫"逻辑还成立吗"——它是判断的投影,非单独创作的区。

**三个主动介入(对话里收起,案卷里高亮)**:

| 介入         | 对话流(收起)                       | 内容与边界                                                                                                                         |
| ------------ | ---------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| **跟踪提醒** | alert chip                         | **只推支点层变化,不推盘面**;一句话:哪个支点/从→到/含义/建议复查。MVP 降级为"打开时主动复查",后台 push 留 v2                        |
| **决策刹车** | `decision` 对账块(need_input 交互) | 文字列「当初支点 vs 现在」+ **强制理由** + **情绪型理由软拦**("要不要再想想",可 override,照实记录)。**质询一致性不评方向(非荐股)** |
| **复盘对账** | 复盘对话 + review chip             | 逐个行动取 ref_snapshot 反推(对过程不对结果)→ 四象限「行动顺逻辑×结果」→ 能力/运气 → lesson 反哺                                   |

**Talor 落地**:卡是**数据驱动的独立渲染**,**不走 talor block**(LLM 只用 card 工具写,不 emit 卡 block)。对话流内嵌卡(系统从 card 工具调用结果提取 card_id → 独立 `StockCardInline` 组件 chip/展开,读 card-repo)+ 右侧常驻案卷面板(独立读 card-repo:两区 + 支点视图 + 时间线时光机)。决策对账 = 案卷内独立交互表单(收集理由 → 调 `append_decision`),非 block。详见 `card-ui-rendering-plan.md`。**合规**:全程非荐股,教练式提醒、克制高信噪。

## §11 纪律与边界(代码强制的硬规则)

- **铁律信号必经 compute 工具**:排雷/分位/逆向DCF/硬证伪只能由 6 个 `compute_*`(invest_lib)产出;**禁止 agent 在脚本里自写阈值/公式**(推理模板/prompt 明令 + 快照记录调用来源,违规可审计)。计算逻辑在代码,LLM 中转的是数据值不是规则。
- **数据经 LLM 的取舍**:数值由 MCP 经 LLM 喂给 compute 工具——标量/小入参稳;长序列(60+点)有 token/转写风险,留 `series_ref` 引用作 fallback。
- **判断必挂支点、支点必指向事实**:每条判断 `fulcrums` 非空,每个支点 `ref` 指向事实区一条事实且带 trigger;否则拒绝落库。
- **决策必注明理由**:`decision.reason` 必填,无理由拒绝落库。
- **快照不可变**:复查产出新 snapshot,不改旧;时间线 append-only;脚本与调用存进快照 = 可复现 provenance。
- **盯支点不盯盘面**:个股资金流/情绪 v2 才接,且只做择时/风险,不进核心判断。
- **事实必溯源**:每条 Fact 带 source;硬数据关键字段多源 `cross_check`,存疑标记不直接采信。
- **软数据来源分级**:检索只采 **T1-T3**(官方/研报/主流财经,见 §8.6),T4↓ 不采;一手优先、承重软支点 `corroboration≥2`、事实/观点分离、带发布日期;检索工具域名分级 + 卡校验 tier 双层强制。
- **管理一票否决**:治理红线(资金占用/违规/控制人风险)直接弃,不进估值。
- **复盘 point-in-time**:重放时间线,只用当时已知事实。
- **合规**:输出为研究分析与跟踪,非荐股;操作由用户决策。

## §12 验收标准(MVP 达成判据)

1. 给定一只 A 股代码,主 agent 委托三专家产出首个 **snapshot**:硬数值由数值 MCP 取、经 `compute_*` 工具算出(带 provenance、多窗口趋势),软定性经 web 检索(**限 T1-T3**)由 agent 读出(带 source.tier + 发布日期)。
2. 每条判断**含 ≥1 支点**,每个支点 **`ref` 指向一条事实** + 变化阈;全卡命门支点 ≥3(含 ≥1 hard + ≥1 soft)。
3. 复查产出新 snapshot,`delta_vs_prev` 记录 fact_changes;hard 支点破阈被 `check_falsification` **确定性检出**(自动标 `shaken`),**agent 推理裁定 `broken`/回 `holding`**;soft 支点变化能被 agent 复读识别裁定。
4. 支点变化 → 反查所撑判断并重判(计入 rejudges),结论可变可维持。
5. 决策落 **decision**(`reason` 必填,ref snapshot);平仓后复盘 agent 落 **review**,逐行动取 ref_snapshot 反推、产出四象限(行动顺逻辑×结果)归因 + lesson。
6. 同一份输入,两次调 `compute_*` 得到**完全一致**的铁律信号(可复现);invest_lib 单测全绿。
7. 换数值 MCP(配置层)后,compute 工具与卡片结构无需改动即可重新跑通(可替换性验证)。
8. 全程**事实/判断分区清晰**,判断可追溯到支点(再到事实);时间线 append-only,可重放重建任一时刻。
9. **UI**:卡以 chip(对话流)/ 案卷(右侧)两态呈现;时间线可**跳转查看历史快照**(只读 + as_of);决策经**对账块**(强制理由、情绪型软拦)。
10. invest_lib、card-repo 单测全绿;`npm run typecheck` 无新增错误。
